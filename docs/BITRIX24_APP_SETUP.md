# Регистрация приложения в Битрикс24

Как подключить **procure-ai** к порталу Битрикс24 как локальное приложение. Эту часть
нельзя сделать кодом — она выполняется один раз вручную администратором портала в
интерфейсе Битрикс24.

> Термины ниже — это поля из карточки локального приложения в Битрикс24.

## Предусловия

- У вас есть развёрнутый UI-сервис приложения, доступный по HTTPS (далее `<APP_URL>`,
  например `https://procure.example.com`).
- Вы — администратор портала Битрикс24.

## Шаги

1. Войдя **под администратором** портала, откройте **«Разработчикам»** → **«Другое»** →
   **«Локальное приложение»** (точные названия пунктов зависят от версии и тарифа Битрикс24 —
   ищите создание именно *локального* приложения). Для локального приложения «добавить» и
   «установить» — одно и то же действие; первым приложение откроет тот же администратор, и
   именно этот первый запуск покажет страницу установки `/install`.

2. Заполните пути обработчиков:

   | Поле | Значение | Зачем |
   |------|----------|-------|
   | **Путь вашего обработчика** | `<APP_URL>/` | Основной интерфейс. Битрикс24 встроит его в **левое меню** (раздел «Приложения»). |
   | **Путь для первоначальной установки** | `<APP_URL>/install` | Установщик. Битрикс24 показывает его при первом открытии; страница вызывает `installFinish()` и завершает установку. |

3. **«Использует только API»** — **выключено**. У приложения есть пользовательский
   интерфейс, поэтому установку подтверждает страница `/install`, а не callback.

4. Выдайте права (scope), которые нужны приложению:

   - `crm` — работа со сделками;
   - `user_brief` — профиль пользователя;
   - `tasks` — задачи;
   - `entity` — универсальное хранилище данных приложения в Битрикс24.

   > Список соответствует `getRequiredRights()` в `ui/app/composables/useB24.ts` — держите их синхронными.

5. Сохраните приложение.

> **Аутентификация.** Внутри портала фрейм приложения авторизуется автоматически (Битрикс24
> передаёт параметры в JS SDK), поэтому `CLIENT_ID`/`CLIENT_SECRET` локального приложения для
> текущего функционала прописывать **не нужно**. Обращения к CRM из backend/MCP идут через
> **входящий вебхук** (`NUXT_BITRIX24_WEBHOOK_URL`), а не через OAuth локального приложения.

> **Доступ к приложению (важно для локального приложения).** Страницы-обработчики (`/` и
> `/install`) отдаются **без HTTP Basic** — иначе кросс-доменный iframe Битрикс24 не смог бы их
> загрузить и установить приложение. Вместо Basic доступ к API закрыт **сессией приложения**:
> - **Внутри портала** сессия устанавливается автоматически: фронтенд берёт `AUTH_ID` и домен из
>   JS SDK и зовёт `POST /session/b24`, а backend подтверждает портал одним вызовом `app.info`.
>   Пароль внутри портала не спрашивается.
> - **Открыли отдельно в браузере** (вне портала) — приложение показывает встроенную форму входа
>   (логин/пароль из `PUBLIC_PAGE_BASIC_AUTH_USER/PASS`), которая ставит ту же сессию.

> **Self-hosted коробка.** Приложение работает во фрейме портала, поэтому backend отдаёт CSP
> `frame-ancestors` с доменами Bitrix24 (облачные `*.bitrix24.ru/.com/.by` — по умолчанию). Если
> ваш Битрикс24 развёрнут на собственном домене, укажите его origin в `B24_FRAME_ANCESTORS`
> (env приложения, через пробел можно несколько) — иначе браузер заблокирует встраивание во фрейм.
> Эта же переменная задаёт **allowlist доменов для `/session/b24`** (проверка `app.info`): для
> коробки её **обязательно** прописать, иначе вход внутри портала не пройдёт (домен не в allowlist).

## Что происходит после сохранения

- При первом открытии приложения администратором Битрикс24 показывает во фрейме
  страницу `/install` («Идёт установка…»). Она дожидается готовности фрейма и вызывает
  `installFinish()`.
- После этого Битрикс24 считает приложение установленным, перезагружает фрейм и открывает
  основной интерфейс (`/`). Приложение появляется в левом меню для всех сотрудников.

> ⚠️ Пока со страницы `/install` не вызван `installFinish()`, Битрикс24 считает приложение
> «не установленным»: интерфейс заблокирован, а обычные сотрудники видят сообщение
> «обратитесь к администратору, он не завершил установку».

## ⚠️ Обязательно: донастроить схему на свежем портале

Страница `/install` **только подтверждает установку** — поля сделки она не создаёт. А создание
сделки (`procuredeal.create`) пишет в кастомные поля `UF_CRM_DEAL_SH_PRCHS_AI_FILE` (файл
документа) и `UF_CRM_DEAL_DOGOVOR` (договор). Если их на портале нет, сделка создастся, но
**молча потеряет вложенный файл и привязку договора**.

**С версии #176 это происходит автоматически:** страница установки (`/install`) сама вызывает
идемпотентный метод `shef:purchase.api.procureinstall.ensureSchema` (через scope `crm`) ещё до
`installFinish` и показывает итог — «Поля сделки настроены ✓» либо причину (обычно — не выдан
scope `crm`). Повторная установка/вызов безопасны: поле создаётся только если его нет.

Ручной вызов нужен лишь как **фолбэк** (старый портал, где автодонастройка не сработала, — например
не был выдан `crm`):

```bash
curl -X POST 'https://<portal>/rest/<user_id>/<webhook_code>/shef.purchase.api.procureinstall.ensureSchema'
```

…либо через **Браузер REST-запросов** на портале («Разработчикам» → «Другое»). Метод вернёт
`{ ok, created[], existing[], failed[], checklist[] }`; каталог, воронку «Закупки» и реквизит
`RQ_INN` он не создаёт молча, а отдаёт в `checklist` для ручной проверки. Список полей и детали —
[`b24-controller/README.md`](../b24-controller/README.md).

## Проверка

Статус установки приложения — REST-метод `app.info`, поле `INSTALLED` должно быть `true`.
Вызвать можно из «Браузера REST-запросов» на портале или из консоли разработчика в открытом
приложении:

```js
// Актуальный SDK @bitrix24/b24jssdk (НЕ устаревший глобал BX24.callMethod):
const r = await $b24.actions.v2.call.make({ method: 'app.info' })
console.log(r.getData().result.INSTALLED) // true — установка завершена
```
