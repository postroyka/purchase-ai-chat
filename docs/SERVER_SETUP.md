# Подготовка сервера procure-ai

Документ для тех, кто разворачивает и проверяет приложение на сервере.
Простыми словами: что это, что уже сделано, как проверить и что дальше.

---

## 1. О чём работа (коротко)

`procure-ai` — приложение, которое из загруженного прайс-листа поставщика
(PDF / XLSX / DOCX) автоматически создаёт сделку в Bitrix24 (воронка «Закупки»).

Приложение работает в Docker и состоит из нескольких контейнеров:

| Контейнер | Что делает |
|---|---|
| `procure-app` | Сайт + бэкенд: приём файлов, обработка, отчёт |
| `procure-mcp` | Прослойка к Bitrix24 (наружу не торчит) |
| `procure-redis` | Хранилище заданий |
| `procure-watchtower` | Сам обновляет приложение при выходе новой версии |
| `nginx-proxy` + `acme-companion` | Внешний вход + HTTPS-сертификат |

Образы собираются автоматически на GitHub и скачиваются на сервер из
приватного хранилища GHCR. Исходный код на сервер **не** клонируется.

---

## 2. Что сделано на сервере (отчёт)

Сервер — Ubuntu 24.04. Выполнено:

1. Установлены `curl`, `make`, Docker (с правом запуска без `sudo`).
2. Открыты порты 80 и 443, настроен firewall.
3. Скачаны конфиги проекта (`docker-compose.*`, `Makefile`, `.env.prod.example`).
4. Заполнен `.env.prod` (секреты, домен). **Остаётся вписать рабочий вебхук Bitrix24.**
5. Выполнен вход в приватное хранилище образов: `docker login ghcr.io` → `Login Succeeded`.
6. Создана сеть `proxy-net`, запущен `nginx-proxy` + `acme-companion`.
7. Запущен основной стек (`make prod-up`) — все контейнеры поднялись и здоровы.

**Открытый вопрос:** HTTPS-сертификат Let's Encrypt на момент написания ещё
выпускался (acme-companion повторяет попытки). Это нужно подтвердить проверкой
из раздела 3 — сайт по `https://` должен открываться.

## 3. Чек-лист ручной проверки

Отмечайте по порядку. Команды для проверки — в разделе 4 (запускаются скриптом).

- [ ] **Контейнеры запущены.** `docker ps` показывает 6 контейнеров со статусом `Up`
      (`procure-app`, `procure-mcp`, `procure-redis`, `procure-watchtower`,
      `nginx-proxy`, `acme-companion`).
- [ ] **Бэкенд жив.** `app /health` отвечает `{"ok":true,"redis":"ok"}`.
- [ ] **MCP жив.** `mcp /api/health` отвечает.
- [ ] **Redis жив.** `redis-cli ping` отвечает `PONG`.
- [ ] **Сертификат выпущен.** В логах `acme-companion` есть «Certificate deployed»,
      сертификат имеет срок действия.
- [ ] **Сайт открывается снаружи.** `https://purchase.postroyka.by` отвечает
      (`200`, либо `401` — запрос пароля, это нормально).
- [ ] **Вход на страницу работает.** В браузере открывается публичная страница,
      пускает по логину/паролю из `.env.prod`.
- [ ] **(после вебхука Б24)** Загрузка тестового файла создаёт сделку в Bitrix24.

> Пункт «создаёт сделку» зависит от готовности инструментов MCP. На текущем этапе
> возможно, что инструменты — заглушки; тогда проверяется только приём файла и отчёт.

## 4. Скрипты проверки (запустить и прислать вывод)

Скрипты ничего не меняют — только читают состояние и печатают отчёт.
Цель: не набирать команды руками, а запустить одну строку и прислать результат.

### На сервере (Linux/Ubuntu)

```bash
cd ~/procure-ai
bash smoke-test.sh
```

Файл: `scripts/smoke-test.sh`. Проверяет контейнеры, внутренние health-проверки,
сертификат, HTTPS и авторизацию. В конце — строка ИТОГ (сколько OK / FAIL).

> **Важно про NAT.** Сервер за NAT не может достучаться до себя по публичному
> домену. Поэтому скрипт по умолчанию проверяет HTTPS/сертификат через локальный
> nginx-proxy приёмом `curl --resolve <домен>:443:127.0.0.1` — соединение идёт на
> `127.0.0.1`, но имя домена и сертификат проверяются как настоящие. Чтобы
> проверить через обычный DNS (например, с внешней машины), запустите:
> `RESOLVE_IP="" bash smoke-test.sh`. Для самоподписанного сертификата —
> `INSECURE=1 bash smoke-test.sh`.

### С обычного компьютера (Windows)

```powershell
powershell -ExecutionPolicy Bypass -File .\smoke-test.ps1
```

Файл: `scripts/smoke-test.ps1`. Проверяет снаружи: DNS, порт 443, сертификат,
`/health`, главную страницу.

> Как получить скрипт на сервер/компьютер: скопировать файл из папки `scripts/`
> репозитория (или скачать его содержимое и сохранить под тем же именем).

## 5. Что дальше

1. Дождаться/подтвердить выпуск TLS-сертификата и зелёный `https://`
   (если долго `503` в challenge — перезапустить `docker restart acme-companion`).
2. Вписать рабочий вебхук Bitrix24 в `.env.prod` и выполнить `make prod-redeploy`.
3. Прогнать `smoke-test.sh` на сервере и `smoke-test.ps1` с рабочего компьютера,
   пройти чек-лист из раздела 3.
