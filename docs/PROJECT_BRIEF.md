# Бриф проекта — `procure-ai` (Закупки/PST)

> Источник истины — `docs/ТЗ_Закупки_PST.md` (версия 1.10). Этот документ — навигатор по нему.

## Роль

Инженер реализует проект автоматизации обработки предложений поставщиков для ООО «Строительный Берег» (`postroyka.by`). Стек — Bitrix24 + Node.js + Claude Code (headless) + DeepSeek *(DeepSeek подключён как модель-провайдер для Claude Code CLI на сервере — `~/.claude/settings.json`, см. «Статус инфраструктуры»; отдельные `DEEPSEEK_*` переменные из ТЗ в коде не используются)*. Работаем строго по ТЗ. Инкрементальные коммиты, прод не трогаем без явного подтверждения.

## Архитектура — жёсткие границы

Четыре слоя. **Главный принцип: AI-ядро и парсинг ничего не знают о Б24. Любое обращение к учётной системе — ТОЛЬКО через MCP-сервер.** При переходе на 1С УТ меняется только MCP-слой.

| Слой | Контейнер | Содержимое |
|---|---|---|
| UI | app | `bitrix24/templates-dashboard` поверх Bitrix24 UI Kit |
| Backend | app | Node.js REST API, оркестрация `claude code`, журнал заданий, cron-cleanup |
| AI | app | `claude code` headless, промпт из `prompts/main.md` (движок — Claude Code CLI; провайдер модели — Anthropic или DeepSeek через env `ANTHROPIC_*` в контейнере, см. README; `DEEPSEEK_*` из ТЗ в коде не используются) |
| MCP | mcp | Собственный MCP-сервер на базе `bitrix24/templates-mcp` |

### Docker — два контейнера

Архитектура разделена на **два образа**:

- **`app`** — backend (Node.js) + UI (статика) + Claude Code CLI + промпты. Оркестрирует обработку файлов, вызывает MCP через stdio/SSE.
- **`mcp`** — MCP-сервер (`bitrix24/templates-mcp`). Единственное место, которое знает о Б24 и обращается к `b24-controller`. При смене учётной системы — меняется только этот контейнер.

Деление обусловлено архитектурным принципом изоляции: `app` не имеет прямого доступа к Б24 API — только через `mcp`.

**Интеграция с Bitrix24.** MCP-инструменты (`b24_pst_crm_*`) вызывают **REST-контроллеры `shef:purchase.api.procure*`**, добавленные в живой модуль коробки `shef.purchase` (через его `restIntegration`). Транспорт — стандартный REST-вебхук (`NUXT_BITRIX24_WEBHOOK_URL`); отдельного URL контроллера нет. `B24_CONTRACTS_API_URL` убран — не нужен. Исходники — `b24-controller/`, деплой по релизному тегу `v*` (workflow `deploy-b24.yml`) или вручную `make deploy-b24`; rsync только `procure*.php`, без `--delete`. ⚠️ Образы (`deploy.yml`) едут **CD на каждый зелёный main**, а PHP-контроллеры — **только по тегу `v*`**; связанные правки контракта Node↔PHP выкатывать тегом. **Существующий код `shef.purchase` не трогаем.**

## Ключевые бизнес-правила (не менять без согласования)

- **Стадия сделки: `C1:NEW`** (воронка id=1 «Закупки»), через ENV `B24_DEAL_DEFAULT_STAGE_ID`.
- **Дубли сделок не проверяем** — сделка создаётся всегда, даже при повторном импорте того же файла. Инструмента `find_existing_deal` нет. _(Решение клиента, 2026-06-19: для MVP поиск дублей точно НЕ нужен; вернуться к нему **при выходе в тираж** — тогда понадобится, ключ дедупликации согласовать отдельно. Блокер B-2 закрыт как «не требуется сейчас».)_
- **Товарные позиции — спецрежим:**
  - Из документа берём цену **БЕЗ НДС** за 1 единицу и количество.
  - Единицу измерения игнорируем — всегда штуки.
  - В Б24: налог **20%**, флаг **«НДС включён в цену» = Y**, цена за 1 из документа, количество.
  - Это намеренно, не баг — не менять.
- **Валюта: только BYN.** Не-BYN → сделка не создаётся, причина в отчёте. Backend проверяет только формат/размер.
- **Допустимые форматы: PDF, XLSX, DOCX, XLS, JPG/PNG.** _(Обновлено 2026-06-09: к ТЗ добавлены `.xls` и фото/скан. Текст извлекается на сервере — PDF/изображения через OCR (tesseract), office через python-хелпер; структуру из текста по-прежнему достаёт агент-LLM.)_
- **Матчинг товара:** только активный, только родительский. Несколько найдено → активный с минимальным ID. Два артикула поставщика на один товар (R-12) → один хранится; второй не находим → предупреждение, позиция не привязывается, ручная обработка.
- **Поиск поставщика** — по УНП через реквизит `RQ_INN` (страна `crm_requisite_preset_country_id`, default 4 = Беларусь), `CCrmCompany::GetListEx` с фильтром `RQ`. Точное совпадение (`=`). Несколько компаний с одним УНП → берём с **минимальным ID**. Российских (ИНН+КПП без УНП) не ищем.
- **Поиск договора** — контроллер `procurecontract.find`: `CLIENT=CO_<supplierId>`, `ACTIVE=Y`, `STATUS≠TO_DELETE` (брак), `TYPE ∈ {PURCHASE, PURCHASE_ZAK}`; опционально точное совпадение по `NUMBER` и `DATE` (d.m.Y). Несколько → минимальный ID. Детали — `b24-controller/IMPLEMENTATION_NOTES.md` §B3.
- **Отчёт пофайловый.** `GET /job/:id/status` отдаёт по файлам `status`, `result`, `error` и (issue #192) `problem` — человекочитаемую причину, если сделка не создана (бизнес-ошибка/нераспознанный документ). UI показывает «Без сделки» + причину, а не зелёное «Готово». При `SHOW_TIMINGS=true` дополнительно отдаются тайминги по файлам (`startedAt/agentMs/durationMs/extractMethod/speed`) + флаг `showTimings` — для живого mm:ss, замеров и класса «быстро/медленно» в логе (не в метрики).
- **`create_deal`** прикрепляет исходный файл к карточке сделки и пишет лог обработки в комментарий.
- **Секреты — только в `.env.prod` на сервере.** Не коммитить, не запекать в образ.
- **Реальные инвойсы не коммитить** — держать как фикстуры вне репозитория либо обезличенные.

| `FEEDBACK_RATE_LIMIT_MAX` / `_WINDOW_MS` | антиспам отзывов (умолч. `0` = выкл. / `3600000`). Считаются попытки; `>0` включает лимит. |

## Правила работы

- Читать ТЗ целиком перед первым шагом.
- Маленькие осмысленные коммиты.
- **Спрашивать подтверждение** перед: операциями на прод-сервере, удалением данных, изменением прав доступа, коммитом секретов или реальных данных клиента.
- Не коммитить `.env.prod`, ключи, PAT, реальные инвойсы.
- Тест-файлы (реальные инвойсы) — в Google Drive, локально в gitignore-папке фикстур.
- Если бизнес-правило кажется «неправильным» — не менять, спросить.
- Документировать по ходу: README, как править промпт, как откатываться, как читать журнал заданий.
