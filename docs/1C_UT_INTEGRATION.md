# ТЗ: интеграция с «1С: Управление торговлей» через MCP

`Последняя ревизия: 2026-06-20` · Статус: черновик для 1С-программиста (форматы устоялись на Bitrix24-реализации)

## 1. Контекст в двух абзацах

Система `procure-ai` принимает счёт поставщика (PDF/скан/Excel), ИИ-агент (Claude) распознаёт его и
**создаёт документ закупки** в учётной системе. Агент общается с учётной системой **только** через 4
инструмента MCP-сервера (Model Context Protocol). Сегодня эти 4 инструмента ходят в Bitrix24 REST
(`shef:purchase.api.*`). Задача — чтобы те же 4 инструмента ходили в **1С: УТ** по аналогичному API.

**Что нужно от 1С:** поднять HTTP-сервис (REST/JSON) с **4 методами**, повторяющими контракт ниже
(вход/выход байт-в-байт по смыслу). Логика агента и MCP-сервера не меняется — меняется только адресат
вызова. То есть это «адаптер»: MCP-инструмент → HTTP-сервис 1С → данные 1С.

```
[Агент Claude] → [MCP-инструмент] → (сейчас) Bitrix24 REST shef:purchase.api.*
                                   → (цель)  HTTP-сервис 1С: УТ
```

## 2. Контракт операций (4 метода)

Везде JSON. «Не найдено» — это **не ошибка**: возвращайте объект с `id: null` (агент сам решит, что делать).
Реальная ошибка (нет связи/исключение) — HTTP 4xx/5xx с телом `{ "error": true, "code": "...", "message": "..." }`.

### 2.1. Найти поставщика по УНП — `findSupplierByUnp`

| | |
|---|---|
| Назначение | Найти контрагента-поставщика по УНП (белорусский 9-значный номер плательщика) |
| Вход | `{ "unp": "123456789" }` — ровно 9 цифр (пробелы/дефисы уже убраны на стороне MCP) |
| Выход (найдено) | `{ "id": 4021, "title": "ООО Ромашка", "unp": "123456789" }` |
| Выход (не найдено) | `{ "id": null }` |

Правила матчинга (как в B24):
- Поиск по реквизиту **ИНН/УНП контрагента**, страна = Беларусь, **точное** совпадение.
- Если несколько контрагентов с одним УНП — вернуть с **минимальным id** (детерминированно).
- **Вне scope:** российские поставщики (ИНН+КПП без УНП) — не ищем.
- `id` — внутренний идентификатор контрагента в 1С (строкой или числом — фиксируйте тип и держитесь его).

### 2.2. Найти договор поставщика — `findContract`

| | |
|---|---|
| Назначение | Найти **активный** договор закупки с данным поставщиком |
| Вход | `{ "supplierId": "4021", "number": "243Э20", "date": "15.03.2025" }` — `number`/`date` опциональны |
| Выход (найдено) | `{ "id": 88, "number": "243Э20", "date": "15.03.2025" }` |
| Выход (не найдено) | `{ "id": null }` |

Правила матчинга:
- Договор привязан к контрагенту `supplierId`, **активен**, вид договора **«С поставщиком»** (в B24: TYPE ∈
  {Закупки, Закупки-Комиссионный}), статус ≠ «Брак»/аннулирован.
- `number` — **гомоглиф-устойчивое** сравнение: латиница/кириллица-двойники считаются одинаковыми
  (`243Э20` с лат. «E» = с кир. «Э»). Если `number` не передан — не фильтруем по номеру.
- `date` — **точное** совпадение в формате `d.m.Y`. Если не передан — не фильтруем по дате.
- Несколько совпадений → **минимальный id**.
- `date` в ответе — дата договора, формат `d.m.Y`.

> Договор **обязателен** для создания документа закупки: если агент не нашёл договор, он останавливается и
> сделку не создаёт. Поэтому корректный матчинг договоров критичен.

### 2.3. Найти номенклатуру по артикулу — `findProductByVendorCode`

| | |
|---|---|
| Назначение | Найти позицию каталога (номенклатуру) по артикулу поставщика |
| Вход | `{ "vendorCode": "тех 100х25х6000" }` |
| Выход (найдено) | `{ "id": 9100, "name": "Доска 100×25×6000", "vendorCode": "тех 100х25х6000" }` |
| Выход (не найдено) | `{ "id": null }` |

Правила матчинга:
- Поиск по полю **артикул** (в B24: `PURCHASE_ARTICLE` у активной «родительской» номенклатуры).
- **Гомоглиф-устойчиво** (лат./кир. двойники = одно). Несколько → **минимальный id**.
- **Только по артикулу.** Поиск по наименованию намеренно НЕ поддерживается (бизнес-решение).
- Если артикула в каталоге нет — это нормально (`id: null`): позиция уйдёт в документ без привязки к
  номенклатуре / в лог (см. 2.4).

### 2.4. Создать документ закупки — `createDeal`

| | |
|---|---|
| Назначение | Создать документ закупки у поставщика и приложить исходный счёт |
| Вход | см. ниже |
| Выход | `{ "dealId": 31840 }` (идентификатор созданного документа) |

Тело запроса:

```jsonc
{
  "supplierId": "4021",                 // контрагент-поставщик (из 2.1)
  "contractId": "88",                   // договор (из 2.2) — ОБЯЗАТЕЛЕН
  "responsibleUserId": "20",            // ответственный (менеджер) — id пользователя 1С
  "fileName": "schet_243.pdf",          // имя исходного файла
  "fileContent": "<base64>",            // содержимое файла, base64 (вложение к документу)
  "documentDate": "15.03.2025",         // дата документа (d.m.Y), опц. → дата документа; нет → текущая
  "processingLog": "Распознано 7 позиций…", // текст лога обработки → в комментарий/примечание документа
  "items": [
    {
      "productId": "9100",              // номенклатура (из 2.3), опц.: нет → позиция без привязки
      "vendorCode": "тех 100х25х6000",  // артикул из документа, опц.
      "name": "Доска 100×25×6000",      // наименование из документа (обязательно)
      "priceExclVat": 12.99,            // цена за единицу БЕЗ НДС (2 знака)
      "quantity": 5                     // количество, целое (единица всегда «шт»)
    }
  ]
}
```

Бизнес-правила документа (как в B24, перенести на модель 1С):
- **Валюта — BYN.** Налог **НДС 20%**, цена указана **без НДС**, но в документе НДС **включается сверху**
  (`priceExclVat` × 1.20). Это намеренное бизнес-правило, не ошибка.
- **Единица измерения — всегда «шт».** `quantity` — целое.
- **Документ создаётся всегда**, без проверки дублей.
- `items` **может быть пустым**: если ни одна позиция не сопоставлена (все артикулы не найдены), документ
  создаётся **без табличной части**, а позиции уходят текстом в `processingLog` (агент так и сделает —
  передаст пустой `items` + лог).
- `priceExclVat` уже округлён до 2 знаков на стороне MCP; примите как есть.
- `documentDate` → дата документа (в B24 — `BEGINDATE` сделки). Непарсибельна/нет → текущая дата.
- `fileContent` (base64) приложить как файл к документу. Ограничение размера — ~25 МБ (см. MAX_ATTACH).
- `processingLog` записать в примечание/комментарий документа (в B24 — поле COMMENTS + запись в таймлайн).
- В ответе вернуть **id созданного документа** как `dealId`.

## 3. Сводка форматов («устаканились»)

| Поле | Формат | Примечание |
|---|---|---|
| УНП | 9 цифр | пробелы/дефисы убираются ДО вызова |
| Даты | `d.m.Y` | напр. `15.03.2025` |
| Валюта | BYN | фиксировано |
| Цена | `priceExclVat` — за единицу, **без НДС**, 2 знака | в документе НДС 20% сверху |
| Количество | целое | единица всегда «шт» |
| Артикул / № договора | строка, **гомоглиф-устойчивое** сравнение | лат./кир. двойники = одно |
| Тай-брейк | **минимальный id** | при нескольких совпадениях |
| Файл | base64 | вложение к документу |
| id | строка или число | зафиксировать тип и держаться его во всех 4 методах |

## 4. Маппинг сущностей: Bitrix24 → 1С: УТ

| Сущность | Bitrix24 (сейчас) | 1С: УТ (предложение — уточнить) |
|---|---|---|
| Поставщик | Компания, реквизит `RQ_INN`, страна BY | **Контрагент** (поиск по ИНН/УНП) |
| Договор | Список «Договора» (iblock 32), TYPE Закупки | **Договор с поставщиком** |
| Номенклатура | Каталог, `PURCHASE_ARTICLE` | **Номенклатура**, реквизит «Артикул» |
| Документ закупки | Сделка (воронка «Закупки», стадия `C1:NEW`) | **Заказ поставщику** / **Поступление товаров** (выбрать) |
| Ответственный | Пользователь B24 | **Пользователь/менеджер 1С** |

## 5. Транспорт и авторизация

- **HTTP-сервис 1С** (REST/JSON). 4 метода — отдельные URL или один с полем `method`; согласуем при старте.
- Авторизация: токен (Bearer) либо basic — на ваше усмотрение, MCP-сторона подставит заголовок (сейчас это
  делает `callV2` для вебхука B24, заменяется на вызов 1С).
- Таймауты/повторы: операции должны быть быстрыми (поиск — десятки мс). Повтор транзиентных сбоев — на
  стороне вызывающего; ваши методы должны быть **идемпотентны для поиска**; `createDeal` — нет (создаёт).
- Кодировка UTF-8. Числа — точкой-разделителем.

## 6. Что вне scope (не реализуем сейчас)

- Российские поставщики без УНП (ИНН+КПП).
- Поиск номенклатуры по наименованию (только по артикулу).
- Дедупликация документов закупки (создаём всегда).
- OAuth/мультиарендность (одна база 1С на инсталляцию).

## 7. Открытые вопросы к заказчику / 1С (решить до реализации)

1. **Какой документ 1С** соответствует «сделке закупки» — «Заказ поставщику», «Поступление товаров и услуг»
   или иной? От этого зависит набор полей в `createDeal`.
2. Тип `id` (строка/число) и его стабильность (внутренний код, GUID, номер?).
3. Где хранить **вложение** (присоединённый файл) и **лог обработки** в выбранном документе 1С.
4. Маппинг `responsibleUserId` (id пользователя 1С) и прав на создание документа.
5. Нужны ли статусы/проведение документа сразу, или черновик.
6. Подтвердить правило **НДС сверху** (цена без НДС → +20%) на стороне 1С.

## 8. Приёмка

- 4 метода отвечают по контракту раздела 2 (вкл. `{id:null}` на «не найдено»).
- Сквозной тест: счёт с известным поставщиком/договором/артикулами → создаётся документ с корректными
  позициями, ценами (НДС 20% сверху), вложением и логом; `dealId` возвращается.
- Негатив: неизвестный УНП → `{id:null}`; нет договора → агент останавливается (документ не создаётся);
  неизвестные артикулы → документ без позиций + лог.
