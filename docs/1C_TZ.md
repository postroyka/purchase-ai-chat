# ТЗ для 1С-программиста (кратко)

`Аудитория: 1С-программист / руководитель интеграции.` Это **краткое ТЗ**. **Полный контракт** (4 метода, поля, примеры запрос/ответ, краевые случаи) — [1C_UT_INTEGRATION.md](1C_UT_INTEGRATION.md). При расхождении — приоритет у полного документа.

## 1. Цель
Сейчас ИИ-агент распознаёт счёт поставщика и создаёт **документ закупки в Bitrix24** (через 4 REST-метода `shef:purchase.api.*`). Нужно, чтобы те же 4 операции выполнялись в **«1С: Управление торговлей»**.

Работа делится на **две независимые части**:
1. **Сторона 1С (ваша задача).** 1С выставляет **HTTP-сервис (REST/JSON)** с 4 методами по контракту ниже. Можно вести **автономно**.
2. Сторона нашего MCP-сервера (наша задача) — добавить путь вызова в 1С и переключатель бэкенда. Логика агента не меняется.

## 2. Что реализовать на стороне 1С (HTTP-сервис, JSON UTF-8, числа с точкой)
Соглашение: **«не найдено» — НЕ ошибка** → `{ "id": null }`. Реальная ошибка → `{ "error": true, "code": "...", "message": "..." }`.

| Метод | Вход | Выход (найдено) |
|---|---|---|
| `findSupplierByUnp` | `{ "unp": "123456789" }` | `{ "id, title, unp }` — контрагент РБ по УНП (9 цифр, точное «=») |
| `findContract` | `{ "supplierId", "number"?, "date"? }` | `{ "id, number, date }` — активный договор закупки поставщика |
| `findProductByVendorCode` | `{ "vendorCode": "…" }` | `{ "id, name, vendorCode }` — номенклатура по артикулу (**строго точное** совпадение) |
| `createDeal` | поставщик, договор, ответственный, позиции, `fileName`+`fileContent` (base64), лог | `{ "dealId" }` (+ `warnings[]`) |

**Важно по матчингу** (детали — полное ТЗ §2–§3):
- Поставщик — по реквизиту **УНП**, страна **Беларусь**, точное «=»; несколько → минимальный id.
- Договор — **активный**, вид ∈ {закупка, закупка-комиссионный}, не удалён; номер сверяется **гомоглиф-устойчиво** (латиница/кириллица) со срезом хвостовой пометки в скобках. **Договор обязателен** — без него документ не создаётся.
- Артикул — **только точное совпадение** «как есть» (без гомоглифов и LIKE).

## 3. Документ закупки — поля позиции (НДС-модель #326)
Портал-приёмник настроен «**цены содержат НДС**». На каждую позицию:
- цена за единицу — **1-в-1 со счётом** (поле «с НДС» строки);
- ставка НДС **20%**, единица — шт (до конвертации единиц, см. [UNIT_CONVERSION.md](UNIT_CONVERSION.md));
- **«Итого» = цена × количество**, **«НДС» = Итого × 20/120** (НДС **включён** в Итого);
- эти суммы должны попадать в документ **как переданы** (без пересчёта учётной системой) — чтобы документ совпадал с бумажным счётом до копейки.
> На стороне Bitrix24 это сделано кастомной правкой ядра (`CCrmProductRow::SaveRows` уважает переданные «Итого»/«НДС»). В 1С — обеспечить эквивалент: документ должен принимать готовые суммы строки и не перетирать их собственным расчётом НДС.

**Файлы:** 1С получает только `fileName` + `fileContent` (base64) и кладёт во вложение документа. К файловому хранилищу приложения 1С **не обращается**.

## 4. Приёмка
- 4 метода отвечают по контракту (вкл. `{ "id": null }` на «не найдено»).
- На тестовом счёте: поставщик/договор/артикулы находятся, документ создаётся, **цена/НДС/итог совпадают со счётом** (прод-сверка: `3.66 × 12 → Итого 43.92 / НДС 7.32`; `0.19 × 200 → 38.00 / 6.33`).
- Ошибки связи/исключения возвращаются телом `{ "error": true, … }`, а не падением.
