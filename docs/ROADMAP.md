# План дальнейшей работы (на русском)

`Обновлено: 2026-06-21`

Короткий понятный план: что уже сделано, что делаем дальше и что мешает запуску. Технические детали — в профильных доках (ссылки по тексту). Это «карта», а не ТЗ.

## 1. Цель проекта

Менеджер **не вбивает счёт руками**: загрузил файл прайса/счёта (PDF, Excel, Word, фото/скан) → система сама создаёт сделку в Битрикс24 (воронка «Закупки») с поставщиком, договором, позициями, ценами и вложением. Экономия времени на каждой закупке.

## 2. Где мы сейчас

- **Работает в проде** (`purchase.postroyka.by`): загрузка файла через веб-страницу → разбор агентом → сделка в Б24. Дашборд `/metrics`, обратная связь 👍/👎 по каждому файлу, живой таймер обработки.
- **Деплой автоматический**: зелёный CI на `main` → образ `:latest` → Watchtower подтягивает (~5 мин). Подробности сопровождения — [docs/OPERATIONS.md](OPERATIONS.md).
- **Чат-бот Битрикс24** — есть **каркас** (логика, эндпоинт событий, безопасность, тесты), но **не включён**: нужна подсистема токена + проводка установки + ручная проверка на тестовом портале (см. §4.1 и [docs/B24_BOT.md](B24_BOT.md)).

## 3. Что сделано в этой итерации (июнь 2026)

## 4. Что дальше (по приоритету)

### 4.1. Чат-бот Битрикс24 — довести до запуска (самое крупное) — issue #217

Идея: сотрудник **бросает файл в чат с ботом** → бот разбирает → присылает результат + кнопки 👍/👎. Это альтернатива веб-странице. Полный дизайн и план ручного теста — [docs/B24_BOT.md](B24_BOT.md).

Осталось (что **можно** закодить, оставив только портал-проверку):

1. **Подсистема захвата токена приложения** (длинный шест).
   - Сервер сейчас не хранит `application_token`, без него нельзя проверять подлинность событий бота.
   - Важно (выяснено по офиц. доке Б24): **iframe-установка токен НЕ отдаёт** (отдаёт только `AUTH_ID`/`member_id`/`DOMAIN`); `application_token` приходит **только в серверном событии `ONAPPINSTALL`** (и в каждом последующем событии). ⚠️ Это **уточняет** дизайн в [docs/B24_BOT.md](B24_BOT.md): §2.1 уже дополнен этим примечанием; старый вариант §4 про «клиентский POST токена» заменяется путём `ONAPPINSTALL` (у клиента токена нет).
   - Делаем: Redis-«стор приложения» (`application_token`, `BOT_ID`, домен портала, ключ — `member_id`) + публичный эндпоинт приёма `ONAPPINSTALL`/`ONAPPUNINSTALL`.
   - **Безопасность захвата (важно!):** одного allowlist доменов **недостаточно** — он ничего не аутентифицирует (любой может прислать POST с `auth.domain` из allowlist). По офиц. рекомендации Б24: `ONAPPINSTALL` несёт `application_token` **вместе с access_token установившего пользователя** — нужно **сначала проверить этот access_token на портале** (вызов `app.info` по SSRF-allowlist — ровно как уже делает `/session/b24` в `backend/auth.js`), и только потом доверять токену и писать его в стор по `member_id`. Плюс маскированное логирование.
   - Эндпоинт `/b24/bot/event` начинает сверять токен со стором (constant-time, с фолбэком на `B24_BOT_APPLICATION_TOKEN` для одно-портальной установки).
   - **Что тестируемо без портала:** разбор события, хранение/сверка токена (constant-time), роутинг — юнит-тесты на mock-payload. **Что только на портале:** что access_token из `ONAPPINSTALL` реально валиден через `app.info`, фактическая форма `auth`-блока, и что `ONAPPUNINSTALL` приходит так, как ожидаем. То есть подсистема — **не** «полностью без портала»: модель доверия подтверждается портал-QA (как и `imbot.v2.*` в §3 ниже).

2. **Клиентская проводка установки** (код пишется сейчас, исполняется на портале):
   - при установке вызвать `imbot.v2.Bot.register` (бот) + `imbot.command.register` (команда `feedback` для кнопок 👍/👎);
   - добавить scope **`imbot`** в `getRequiredRights()` (`ui/app/composables/useB24.ts`: сейчас `user_brief, crm, tasks, entity`) и в карточку приложения.
   - ⚠️ **Порядок важен:** scope `imbot` + **переустановка** приложения с этим правом должны пройти ПЕРВЫМИ — без переустановки ни регистрация бота, ни приём `ONAPPINSTALL` (шаг 1) не сработают. То есть шаг 1 (стор/эндпоинт) пишем заранее, но «оживёт» он только после переустановки с `imbot`.

3. **Ручная проверка на ТЕСТОВОМ Битрикс24** (это и есть «оставить только ручную проверку») — сценарии в [docs/B24_BOT.md §12](B24_BOT.md): бросить счёт → получить сделку + 👍/👎; несколько файлов; сообщение без файла; кривой документ; клик 👍/👎 → метрика растёт; неверный токен → 403. Потом переключение на боевой портал — §13 там же.

Вариант «спрятать за `SHOW_TIMINGS`» — лишь один из, тоже часть решения, а не данность.

## 7. Заметка на будущее: обновления зависимостей

В окружении (на раннере CI **и** локально) включена политика pnpm **`minimumReleaseAge`** — она **отклоняет** пакеты, опубликованные меньше ~суток назад (защита от свежих скомпрометированных версий supply-chain). Важно: это **глобальный конфиг pnpm в окружении, его НЕТ в репозитории** (не ищите `.npmrc`/`pnpm-workspace.yaml` в дереве — там его нет). Что это значит на практике:

- если `pnpm up` подтянул совсем свежую версию — CI её **завернёт** на шаге `pnpm install --frozen-lockfile` с ошибкой `ERR_PNPM_MINIMUM_RELEASE_AGE_VIOLATION` (так и было с `uuid@14.0.1` в PR #223);
- решение: **подождать**, пока версия «отлежится» (по умолчанию — так, не обходим политику ради рутинного апдейта). Если версия нужна срочно — pnpm при `pnpm up` сам предлагает добавить её в `minimumReleaseAgeExclude` (пишет per-package `pnpm-workspace.yaml`); это **осознанный обход** для конкретного пакета, не для рутины.
