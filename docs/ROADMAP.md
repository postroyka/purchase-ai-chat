# План дальнейшей работы (на русском)

`Обновлено: 2026-06-21`

Короткий понятный план: что уже сделано, что делаем дальше и что мешает запуску. Технические детали — в профильных доках (ссылки по тексту). Это «карта», а не ТЗ.

## 1. Цель проекта

Менеджер **не вбивает счёт руками**: загрузил файл прайса/счёта (PDF, Excel, Word, фото/скан) → система сама создаёт сделку в Битрикс24 (воронка «Закупки») с поставщиком, договором, позициями, ценами и вложением. Экономия времени на каждой закупке.

## 2. Где мы сейчас

- **Работает в проде** (`purchase.postroyka.by`): загрузка файла через веб-страницу → разбор агентом → сделка в Б24. Дашборд `/metrics`, обратная связь 👍/👎 по каждому файлу, живой таймер обработки.
- **Деплой автоматический**: зелёный CI на `main` → образ `:latest` → Watchtower подтягивает (~5 мин). Подробности сопровождения — [docs/OPERATIONS.md](OPERATIONS.md).
- **Чат-бот Битрикс24** — есть **каркас** (логика, эндпоинт событий, безопасность, тесты), но **не включён**: нужна подсистема токена + проводка установки + ручная проверка на тестовом портале (см. §4.1 и [docs/B24_BOT.md](B24_BOT.md)).

## 3. Что сделано в этой итерации (июнь 2026)

- **Обновления зависимостей** (PR #223): backend/ui/mcp подняты на свежие версии, проходящие политику безопасности CI. Два обновления отложены сознательно (см. §5).
- **Безопасность бота — MIME-проверка** (#216, PR #224): файл из чата проверяется по «магическим байтам» так же, как веб-загрузка (общий `backend/file-validation.js`).
- **Полировка страницы результата** (#201/#203/#221, PR #225):
  - единый формат денег (как на `/metrics`);
  - живой таймер «обрабатывается N сек» — теперь **всегда** виден (успокаивает на медленных файлах);
  - компактный виджет отзыва (поле комментария раскрывается после выбора оценки);
  - машинный код исхода рядом с причиной (для разбора).

## 4. Что дальше (по приоритету)

### 4.1. Чат-бот Битрикс24 — довести до запуска (самое крупное) — issue #217

Идея: сотрудник **бросает файл в чат с ботом** → бот разбирает → присылает результат + кнопки 👍/👎. Это альтернатива веб-странице. Полный дизайн и план ручного теста — [docs/B24_BOT.md](B24_BOT.md).

Осталось (что **можно** закодить, оставив только портал-проверку):

1. **Подсистема захвата токена приложения** (длинный шест).
   - Сервер сейчас не хранит `application_token`, без него нельзя проверять подлинность событий бота.
   - Важно (выяснено по офиц. доке Б24): **iframe-установка токен НЕ отдаёт**; токен приходит **только в серверном событии `ONAPPINSTALL`** (и в каждом последующем событии).
   - Делаем: Redis-«стор приложения» (`application_token`, `BOT_ID`, домен портала по `member_id`) + публичный эндпоинт приёма `ONAPPINSTALL`/`ONAPPUNINSTALL` (захват токена с гардом по allowlist доменов портала, маскированное логирование). Эндпоинт `/b24/bot/event` начинает сверять токен со стором (с фолбэком на `B24_BOT_APPLICATION_TOKEN` для одно-портальной установки).
   - Всё это **тестируется юнит-тестами без портала**.

2. **Клиентская проводка установки** (код пишется сейчас, исполняется на портале):
   - при установке вызвать `imbot.v2.Bot.register` (бот) + `imbot.command.register` (команда `feedback` для кнопок 👍/👎);
   - добавить scope **`imbot`** в `getRequiredRights()` (`ui/app/composables/useB24.ts`) и в карточку приложения.

3. **Ручная проверка на ТЕСТОВОМ Битрикс24** (это и есть «оставить только ручную проверку») — сценарии в [docs/B24_BOT.md §12](B24_BOT.md): бросить счёт → получить сделку + 👍/👎; несколько файлов; сообщение без файла; кривой документ; клик 👍/👎 → метрика растёт; неверный токен → 403. Потом переключение на боевой портал — §13 там же.

### 4.2. Мелкие доработки (есть issue, делаются по схеме PR → ревью → мерж)

- **#205** — форс тестового отзыва агента на **один** аплоад (`?forceFeedback=1` на `POST /upload`) вместо глобального env-флага: тестировать канал без рестарта и без шума в метриках у всех.
- **#207** — **распределение** «быстро/норма/медленно» на `/metrics` (напр. «62% быстро / 30% норма / 8% медленно»). Дёшево: данные уже текут в метрики, пороги есть. *Решение заказчика:* показывать ли распределение в метриках (раньше договаривались «тайминги не в метрики»; распределение — это агрегат, не сырые числа). По умолчанию спрячем за `SHOW_TIMINGS`.
- **#203.2** — точное **время извлечения текста** (мерить вокруг extract в `agent-runner.js`, прокидывать `extractMs`), вместо показа только метода (`ocr`/`pdftotext`).

### 4.3. Анализ скорости — issue #222

Где агент теряет время: «думает» vs «ходит в инструменты». Инструментировать тайминги MCP-вызовов (`mcp-overlay`) и/или разбить поток `stream-json` на размышление/поиск. Это диагностика под жалобу «медленно» — поможет прицельно ускорять.

## 5. Что мешает запуску (и чего НЕ мешает)

- **Веб-сценарий уже запущен** — ничего не мешает, работает в проде.
- **Чат-бот** — мешает только то, что §4.1 требует **ручной проверки на портале** (события/скачивание/отправку нельзя проверить без живого Б24). Код подсистемы и проводки пишется и тестируется заранее.

## 7. Заметка на будущее: обновления зависимостей

В окружении (локально и в CI) включена политика **`minimumReleaseAge`** — она **отклоняет** пакеты, опубликованные меньше ~суток назад (защита от свежих скомпрометированных версий supply-chain). Поэтому:

- если `pnpm up` подтянул совсем свежую версию — CI её **завернёт** (как было с `uuid@14.0.1`);
- решение: либо подождать, пока версия «отлежится», либо (осознанно) добавить её в `minimumReleaseAgeExclude` (`pnpm-workspace.yaml`). По умолчанию — **ждать**, не обходить политику ради рутинного апдейта.
