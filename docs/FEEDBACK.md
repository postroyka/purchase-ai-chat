# Обратная связь от сотрудника (`POST /feedback`)

`Последняя ревизия: 2026-06-20`

Это **первый из трёх каналов** обратной связи из issue **#182** — канал «**сотрудник**». Человек, который
обработал прайс-лист, прямо на странице результата (под КАЖДЫМ файлом, #218) может оценить итог (👍 / 👎) и оставить
комментарий. Отзыв уходит на бэкенд и превращается в **GitHub issue** в настроенном репозитории.

Цель — собирать *структурированный* сигнал против реальных кейсов (а не анекдоты), чтобы итеративно
чинить два главных провала качества с боевого прогона: **сопоставление артикулов (vendorCode)** и
**определение РФ/РБ‑поставщика**.

> Документ для **операторов** (настройка интеграции) и **мейнтейнеров** (триаж issue). Два других
> канала #182 — **от агента** (агрегация `warnings[]`/`feedback[]` на `/metrics`, см. «Канал агента») и
> **от MCP** («Проблемы матчинга» на `/metrics`, см. «Канал MCP») — уже реализованы.

## Поток

```
UI (страница результата)
   └─ оценка + комментарий
        └─ POST /feedback   (сессия приложения + X-PAI-Auth, или Bearer)
             └─ backend/feedback.js  (санитизация + сборка issue)
                  └─ GitHub REST  POST /repos/<owner>/<repo>/issues
                       └─ issue с метками user-feedback, feedback:<kind>
```

Виджет рисуется только когда задание завершилось (`done`/`error`) **и** канал настроен на сервере —
UI заранее спрашивает `GET /feedback/config`.

## Конфигурация (ENV)

| Переменная | Обяз. | По умолчанию | Назначение |
|---|---|---|---|
| `GITHUB_FEEDBACK_TOKEN` | да* | — | GitHub‑токен с правом **Issues → Read and write** на целевом репо. Нет токена → канал ВЫКЛЮЧЕН (виджет скрыт, `POST /feedback` → 503). |
| `GITHUB_FEEDBACK_REPO` | нет | `postroyka/purchase-ai-chat` | Куда писать issue, в формате `owner/repo`. Валидируется по `^[A-Za-z0-9_.-]+/[A-Za-z0-9_.-]+$` до запроса. |
| `FEEDBACK_RATE_LIMIT_MAX` | нет | `0` | Сколько отзывов с одного клиента за окно (антиспам в наш репозиторий). `0` = выключено (внутренний инструмент); `>0` включает лимит. |
| `FEEDBACK_RATE_LIMIT_WINDOW_MS` | нет | `3600000` (1 ч) | Длина окна rate‑limit. |

\* Обязателен только чтобы **включить** канал. Без него приложение работает как раньше, просто без формы
обратной связи.

### Токен

Рекомендуется **fine‑grained Personal Access Token**, ограниченный ОДНИМ репозиторием
(`GITHUB_FEEDBACK_REPO`), с единственным правом **Issues → Read and write**. Это минимальная привилегия:
утечка токена даёт максимум возможность писать issue в один репо, а не доступ к коду или другим репо.
Классический `repo`‑scope тоже подойдёт, но он шире — избегайте.

Метки `user-feedback`, `feedback:<kind>` GitHub создаёт автоматически при первом отзыве (серым цветом по
умолчанию). Хотите свои цвета — заведите их заранее в Settings → Labels.

### Сетевой доступ

Бэкенд‑контейнеру нужен **исходящий HTTPS к `api.github.com`** (порт 443) — оттуда заводятся issue.
На «коробке» с фильтрацией egress (как наш боевой `b24.postroyka.by`) добавьте этот хост в allowlist
рядом с `api.nbrb.by` и доменом портала (см. `docs/SERVER_SETUP.md`). При **кратком** сбое доступа
`POST /feedback` теперь вернёт `202` и положит отзыв в durable‑outbox (#190) — он дошлётся фоном, как
только GitHub снова доступен; в логах будет только код `NETWORK`/`UPSTREAM` (тело/токен не логируются).
Если egress к `api.github.com` закрыт **насовсем**, issue будут копиться в очереди и в итоге отбрасываться
(лимит попыток) — это самая вероятная причина «отзыв не доходит» на первом запуске, смотрите allowlist.

## Форма issue

Успешная отправка создаёт issue:

- **Заголовок**: `[Обратная связь] <метка типа> · <первая строка комментария>` (обрезается до 120
  символов, hostile‑символы вырезаны), где `<метка типа>` — это `👍 Хорошо` / `👎 Проблема` /
  `💡 Предложение` (не «сырой» ключ `positive`/`problem`/`suggestion`).
- **Метки**: всегда `user-feedback` и `feedback:<kind>`, где `<kind>` ∈ `positive` | `problem` | `suggestion`.
- **Тело**: блок «Контекст» (jobId, имя файла, № сделки, **время обработки**, кто сообщил, версия сборки,
  User‑Agent — только заполненные поля) + комментарий сотрудника внутри `<pre><code>` (HTML экранирован,
  Markdown инертен) + секция **«Лог обработки»** (#237) — что распознал агент по этому файлу.
  Подвал помечает программное происхождение.

> **Лог и время (#237).** Их бэкенд подкладывает в issue **сам**, из журнала заданий по `jobId`+файлу
> (`fileEntry.durationMs` и `result.processingLog` хранятся всегда, независимо от `SHOW_TIMINGS`) — тот
> же лог, что и на странице результата (в issue обрезан **жёстче**: до 4000 символов против 10000 в UI).
> Лог — недоверенный текст агента, поэтому hostile‑stripped + HTML‑escaped + обрезан. Best‑effort:
> если задание истекло (`JOB_TTL_HOURS`) **или** результат файла был слишком большим и обрезан в
> журнале, issue заведётся без секции лога.

## Эндпоинты

### `GET /feedback/config` — открыт

```json
{ "enabled": true }
```

Возвращает ровно один булев флаг — настроен ли канал. **Ни токен, ни репо не раскрываются.** Нужен UI,
чтобы решить, показывать ли виджет.

### `POST /feedback` — требует авторизации (сессия приложения + `X-PAI-Auth`, либо Bearer)

Запрос:

```jsonc
{
  "kind": "problem",                // 'positive' | 'problem' | 'suggestion' (обязательно)
  "comment": "Артикул не найден...",// ОПЦИОНАЛЬНО (#218), до 5000 симв. (длиннее — обрезается)
  "context": {                      // всё опционально; собирается приложением
    "jobId": "abc-123",             // ^[A-Za-z0-9-]{1,64}$, иначе отбрасывается
    "fileName": "price.xlsx",
    "dealId": "42",                 // ^\d{1,12}$, иначе отбрасывается
    "appVersion": "<git sha>"
  }
}
```

Ответ `201` (issue создан сразу):

```json
{ "ok": true, "url": "https://github.com/owner/repo/issues/7", "number": 7 }
```

Ответ `202` (GitHub был кратко недоступен → отзыв в очереди, дослыл фоном — #190):

```json
{ "ok": true, "queued": true }
```

Коды:

| Код | Когда |
|---|---|
| `201` | Issue создан сразу. |
| `202` | GitHub был кратко недоступен → issue положен в **durable‑outbox** (#190) и будет дослан фоном; повторять не нужно (`{ ok:true, queued:true }`). |
| `400` | Неизвестный `kind`. (Пустой `comment` теперь допустим — #218.) |
| `401` | Нет валидной авторизации. |
| `429` | Превышен rate‑limit (см. ENV). |
| `502` | GitHub отклонил по **постоянной** причине — 401/403 (токен) или 404 (репо); такие в outbox не кладём, чтобы оператор заметил. Детали — только в логах (код `UPSTREAM`), без токена. |
| `503` | Канал не настроен: пустой `GITHUB_FEEDBACK_TOKEN` **или** некорректный slug `GITHUB_FEEDBACK_REPO` (не `owner/repo`). |

`url` указывает на issue в **приватном** репо — он не показывается рядовому сотруднику (тот видит только
тост «Спасибо»); это сигнал для приложения/мейнтейнера.

## Безопасность

Модуль `backend/feedback.js` портирован из проверенного `mcp/server/utils/github-feedback.ts` и
сохраняет его свойства:

- **Токен не попадает в логи** — ни сам токен, ни URL запроса, ни тело ответа GitHub не пишутся при
  ошибке (оператор из логов контейнера не восстановит креденшл). Логируется только стабильный код
  ошибки (`NOT_CONFIGURED` / `UPSTREAM` / `NETWORK`).
- **Валидация репо** — slug `owner/repo` проверяется до попадания в путь запроса (значение вроде
  `../../users/x` не уведёт вызов на чужой ресурс).
- **Вырезание hostile‑символов** — C0‑контролы, bidi‑оверрайды (U+202A..U+202E, U+2066..U+2069),
  zero‑width/BOM удаляются: подделанный комментарий не «перевернёт» список issue (Trojan Source) и не
  сломает JSON.
- **Инертный рендер** — свободный текст HTML‑экранируется и оборачивается в `<pre><code>`, так что
  Markdown/HTML в нём не исполняется.
- **Авторизация** — `POST /feedback` за `requireAuth`: только пользователи приложения (cookie‑сессия +
  `X-PAI-Auth`, либо серверный Bearer) могут заводить issue, иначе это открытый спам‑вектор в наш репо.
- **Rate‑limit** — окно считает **попытки**, а не успехи: неудачный вызов GitHub тоже тратит слот, что
  гасит тугие циклы ретраев. Тот же Redis‑лимитер, что у `/upload` (мультиинстансо‑безопасный).
- **Таймаут + `redirect: 'error'`** на вызове GitHub — зависший GitHub не «прибьёт» запрос, а 3xx на
  неожиданный адрес не выполняется (defence in depth).

## Приватность — почему контекст допустим

Целевой репозиторий **приватный на запуске**. Поэтому в issue допустимо складывать контекст задания
(jobId, имя файла, № сделки, кто сообщил) — это ровно тот сигнал, по которому мейнтейнер воспроизводит
кейс «артикул не сопоставился / поставщик не РФ». Имя файла и комментарий могут содержать данные клиента
— это сознательный компромисс для приватного репо.

> ⚠️ **Если репозиторий когда‑либо станет публичным** — пересмотрите это: каждое тело issue станет
> world‑readable (это касается обоих каналов — и «сотрудник», и «агент» #182, оба пишут СЮДА). Тогда либо
> заведите отдельный приватный репо под `GITHUB_FEEDBACK_REPO`, либо урежьте контекст. Не путайте это с
> каналом фидбека **MCP‑шаблона** (upstream `mcp/`, `mcp/docs/FEEDBACK.md`) — тот пишет в *публичный*
> шаблонный репо и потому запрещает любые персональные данные; в этом проекте он вырезан из образа.

**Страховка на старте (#190).** Приватность больше не держится только на доке: при наличии
`GITHUB_FEEDBACK_TOKEN` бэкенд на старте делает `GET /repos/{repo}` и пишет в лог **WARNING**, если репо
оказался публичным (`private:false`) — один клик «Make public» теперь не пройдёт незаметно. Проверка
best‑effort и не блокирует запуск (сетевой сбой → строка «не удалось проверить приватность»).

## Триаж

Оба канала пишут в **один** приватный репо (он же — репозиторий кода), различаются только метками:
issue сотрудников — `user-feedback`, issue агента — `agent-feedback` (плюс `feedback:<kind>` у обоих).
Так как agent‑issue машинные (до 10/час) и будут доминировать в бэклоге, **триажьте через фильтр по
метке**, а не по общему списку: `is:issue is:open label:user-feedback` и `… label:agent-feedback`.

- Новые issue падают с меткой `user-feedback` / `agent-feedback` в открытый бэклог — просматривать
  каждую хотя бы раз в неделю по фильтру метки.
- Перевешивайте на `bug` / `enhancement` / `wontfix` / `duplicate`, но **оставляйте** канальную метку,
  чтобы поток оставался queryable.
- Группируйте по корню (артикулы, РФ/РБ; для агента — конкретный инструмент), а не закрывайте дубли —
  ссылайтесь на исходный issue. (Канал агента уже дедупит на стороне сервера, см. ниже.)
- `jobId` в теле позволяет (пока не сработала ретенция `UPLOADS_RETENTION_DAYS`) поднять исходные файлы и
  результат агента на сервере.
- Объём растёт — рассмотрите ОТДЕЛЬНЫЙ приватный репо под фидбек (env уже это допускает).

## Канал «агент» (issue #182)

ИИ‑агент (`backend/agent-runner.js`, промпт `prompts/main.md`) может вернуть в своём JSON‑результате два
**необязательных** поля; их разбирает `processJob` (`backend/index.js`):

- **`warnings[]`** — коды нештатных, но не фатальных ситуаций по файлу (`no_items_matched`,
  `articles_not_in_catalog`, `items_without_article`, плюс warnings из `create_deal`). Идут **только в
  метрики** — панель «Сигналы качества (агент)» на `/metrics`. Коды пинятся по allowlist (`metrics.js`),
  незнакомые схлопываются в `other` (агент читает недоверенные документы → защита от роста кардинальности).
- **`feedback[]`** — «что мешает в инструментах/промпте и как улучшить» (`{ tool?, kind, note }`,
  `kind` ∈ `positive|problem|suggestion|perf`). Каждый элемент считается в метрику «Обратная связь агента»
  (по типу). `positive`/`problem`/`suggestion` дополнительно заводят **GitHub issue** (метки
  `agent-feedback` + `feedback:<kind>`, дедуп по TTL). В теле issue — блок «Контекст» с per-file данными (#219): инструмент, jobId, имя файла, **сделка** (`#<dealId>`, если создана) и **исход** (`outcome` с расшифровкой, напр. `no_deal (сделка не создана)`) — чтобы было видно, к какому файлу относится отзыв, в т.ч. при форс-тесте `AGENT_FORCE_FEEDBACK` с константной note.
  - `perf` (#279) — **agent-only диагностика скорости**: агент добавляет её **всегда, на каждом файле**
    (если узких мест нет — короткое «обработка штатная») — что замедлило разбор, как ускорить. **GitHub issue НЕ заводит** — на проде
    почти каждый файл «медленный» (per-file note уникальна → дедуп не ловит → спам + выжигание hourly-cap).
    Поэтому perf идёт в **метрику** («Скорость» на `/metrics`) + строку **лога** `[perf-diag]` (`docker logs
    procure-app | grep perf-diag`), **и показывается оператору** на странице результата под таймингами
    («⏱ Диагностика скорости агента») — **всегда, когда агент её прислал** (свёрнутый блок, не привязан к
    `SHOW_TIMINGS`). Скрыть блок можно флагом `HIDE_PERF_NOTE=true` в `.env` — это **только UI-косметика**:
    perf-данные всё равно приходят в JSON ответа `/job/:id/status` (`result.feedback`) и в лог `[perf-diag]`.
    Виджет сотрудника perf не предлагает. **Кому и зачем показываем диагностику** (оператор vs сотрудник,
    дефолты `SHOW_TIMINGS`/`HIDE_PERF_NOTE`, что НЕ должно попадать сотруднику/в задачу B24) —
    зафиксировано в [docs/DIAGNOSTICS_POLICY.md](DIAGNOSTICS_POLICY.md) (#320).

Инструкция агенту — в `prompts/main.md` (раздел «Сигналы и обратная связь агента») плюс по строке в
описании каждого нашего MCP‑инструмента (`mcp-overlay/.../deals/*`): «если инструмент мешает — запиши в
`feedback[]`».

> ⚠️ **Деплой двух образов.** Правки `prompts/main.md` и backend едут с **app‑образом**, а строки‑подсказки
> в описаниях MCP‑инструментов запекаются в **MCP‑образ** на `pnpm build`. Пока MCP‑образ не пересобран и
> не передеплоен, агент не увидит подсказку «пиши в `feedback[]`» — серверная агрегация уже работает, а
> побуждение на стороне промпта инструментов ещё нет. Пересоберите MCP‑образ (чеклист деплоя MCP↔PHP, PR #78).

**Дедуп и кап (`agent-feedback.js`).** Агент работает на потоке файлов, поэтому:

- одно и то же трение (`kind` + `tool` + нормализованный текст) заводит **один** issue за окно
  (TTL 14 дней) — а не по issue на каждый обработанный файл;
- жёсткий **кап 10 issue/час** — бэкстоп против всплеска новых трений и против промпт‑инъекции из
  документа, пытающейся спамить репо (плюс ≤10 элементов `feedback[]` на файл). Дедуп ключуется по
  **санитизированному** `tool` (мусорные значения не плодят «разные» одинаковые issue).

Оба guard‑а предпочитают Redis (как rate‑limiter), иначе in‑memory. Всё **best‑effort**: сбой телеметрии
никогда не валит задание; текст агента недоверенный, поэтому проходит те же hostile‑strip + HTML‑escape.
Канал включается тем же `GITHUB_FEEDBACK_TOKEN` (нет токена → agent‑issue не заводятся, но счётчики
`warnings`/`feedback` в метрику пишутся всё равно).

**Как протестировать канал (пока агент сам молчит).** Реальный `feedback[]` агент присылает только при
трении с инструментами, поэтому панель «Обратная связь агента» обычно пустая. Для проверки — env‑флаг
**`AGENT_FORCE_FEEDBACK=true`** (`backend/agent-runner.js`): на каждом файле, если агент сам не прислал
`feedback`, добавляется один тестовый элемент (`kind: problem`, `tool: force-test`). Эффект: счётчик
«Обратная связь агента» на `/metrics` растёт (виден **без** токена), а при заданном `GITHUB_FEEDBACK_TOKEN`
заводится и GitHub issue. Реальный feedback агента флаг не перетирает. Это диагностика — на проде держать
`false`.

> ⚠️ **Чего ожидать.** Запись константная, поэтому: счётчик «Обратная связь агента» инкрементится на
> **каждый** файл (растёт, пока флаг включён), а **GitHub issue заводится один раз** на окно дедупа
> (14 дней) — повторные файлы дают `reason: 'duplicate'`, а не новый issue. Это проверка **транспорта**
> канала (метрика + issue + панель), а не способности агента самому сформировать валидный `feedback[]`.

## Канал «MCP» — «Проблемы матчинга» (issue #182)

Третий канал — «где матчинг не сходится». Исходы инструментов матчинга MCP (поставщик/договор/артикул)
отражаются в результате агента, поэтому сигнал собирается **в бэкенде из самого результата** (без
хрупкой меж‑процессной телеметрии). На `/metrics` появилась секция **«Проблемы матчинга»**:

- **«Где не сошлось»** — единый рейтинг причин: `supplier_not_found` / `contract_not_found` /
  `foreign_supplier` (РФ) / `unsupported_currency` (из `outcomes`) + `no_items_matched` (из `warnings`).
- **«Поставщики без матча (по УНП)»** — новый срез: какие поставщики чаще всего не находятся
  (`metrics:matching:suppliers`, ключ — УНП). Кардинальность ограничена (cap 300, переполнение → `__other__`);
  УНП недоверенный (из документа) → только цифры, длина 4–16, иначе не пишется. `recordMatching` в
  `metrics.js`, вызов — в `processJob`.
- **«Мультиматчи»** (v2, #195) — сколько ДОКУМЕНТОВ имели мультиматч на каждом шаге (поставщик/договор/товар):
  PHP-контроллер ставит `multi:true` при >1 совпадении, агент переносит шаг в `matching.multiMatches`
  (дедуп по файлу). `metrics:matching:multi`, ключ — шаг (allowlist из 3).
- **«Артикулы без матча»** (v2, #195) — топ `vendorCode`, не найденных в каталоге (`matching.unmatchedArticles`).
  `metrics:matching:articles`; артикул недоверенный → санитизация (`[A-Z0-9._/-]`, длина ≤40) + cap 300 → `__other__`.

Так закрывается приёмка #182 «ранжировать самые частые причины провала (артикулы, РФ/РБ)»: артикулы — в
«Сигналах качества» + экономике (`positionsNoArticlePct`), поставщики — здесь.

> **Источник `positionsNoArticle` (#264).** После #258 позиции БЕЗ артикула не попадают в `items[]`,
> поэтому считать их по `items[]` (как было) давало всегда 0 и инвертировало плитку «Потеря на пустых
> артикулах». Теперь счётчик берётся СТРУКТУРНО из ответа агента — `matching.itemsWithoutArticle`
> (агент заполняет на Шаге 4: позиции, у которых НЕ найден код товара поставщика — определяется
> по-позиционно, не «нет колонки артикула во всём документе», см. `prompts/main.md`), а `positionsNoArticlePct` считается от
> базы «сопоставленные + без артикула». «Потеря» (`lostNoArticleByn`) = счётчик × время-на-позицию ×
> ставка (ручной ввод этих позиций), а не цена товара.

## Единый контракт фидбэка (#190)

Две копии «текст → GitHub issue» (`backend/feedback.js` ESM и `mcp/server/utils/github-feedback.ts` TS)
держат **один** контракт, чтобы не разъезжаться:

- **Виды (`kind`):** `positive` | `problem` | `suggestion`. **Виджет сотрудника — только `positive`/`problem`** (#218); `suggestion` остаётся у канала **агента**/MCP и в метриках (`FEEDBACK_KINDS` 3-значный).
  Раньше MCP‑тул использовал `issue` вместо `problem` — выровнено.
- **Кап текста:** 5000 символов (`MAX_COMMENT_LENGTH` ≡ `MAX_DETAILS_LENGTH`).
- **Метки:** канал различается меткой — `user-feedback` (сотрудник) vs `agent-feedback` (агент/MCP), плюс
  `feedback:<kind>` у всех; это намеренно, не дрейф.

Полное переиспользование кода нетривиально (TS‑в‑MCP vs ESM‑в‑backend) — это осознанный долг; едины
именно значения контракта.
