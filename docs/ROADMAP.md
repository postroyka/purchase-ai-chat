## 1. Цель проекта

Менеджер **не вбивает счёт руками**: загрузил файл прайса/счёта (PDF, Excel, Word, фото/скан) → система сама создаёт сделку в Битрикс24 (воронка «Закупки») с поставщиком, договором, позициями, ценами и вложением. Экономия времени на каждой закупке.

## 2. Где мы сейчас

- **Работает в проде** (`purchase.postroyka.by`): загрузка файла через веб-страницу → разбор агентом → сделка в Б24. Дашборд `/metrics`, обратная связь 👍/👎 по каждому файлу, живой таймер обработки.
- **Деплой автоматический**: зелёный CI на `main` → образ `:latest` → Watchtower подтягивает (~5 мин). Подробности сопровождения — [docs/OPERATIONS.md](OPERATIONS.md).
- **Чат-бот Битрикс24** — **код готов**, в фазе **портал-проверки**. На старом портале заказчика используется **legacy** `imbot.*` (`imbot.v2.*` → 404; v2-код закомментирован для возврата при тираже — #241/#242, трекинг #243). Подсистема захвата токена и проводка установки — **сделаны**; осталась **ручная проверка на портале** (события/скачивание файла/отправка). См. §4.1 и [docs/B24_BOT.md](B24_BOT.md).

## 4. Что дальше (по приоритету)

### 4.1. Чат-бот Битрикс24 — довести до запуска (самое крупное) — issue #217

Идея: сотрудник **бросает файл в чат с ботом** → бот разбирает → присылает результат + кнопки 👍/👎. Это альтернатива веб-странице. Полный дизайн и план ручного теста — [docs/B24_BOT.md](B24_BOT.md).

Осталось (что **можно** закодить, оставив только портал-проверку):

1. ✅ **Подсистема захвата токена приложения (сделано, issue #217).**
   - `backend/app-store.js` (Redis, хранит **sha256-хеш** токена по `member_id` — сам токен не персистится) + публичный эндпоинт `POST /b24/app/event` (`ONAPPINSTALL`/`ONAPPUPDATE`/`ONAPPUNINSTALL`).
   - **Безопасность:** на захвате проверяем `access_token` из события на портале (`app.info` по SSRF-allowlist — как `/session/b24`); одного allowlist домена недостаточно. `ONAPPUNINSTALL` чистит только при совпадении токена.
   - ✅ **Фолбэк-захват из события бота (#241):** если старый портал НЕ шлёт `ONAPPINSTALL` на callback, `application_token` ловится из **первого подлинного события бота** — подлинность через `app.info` + домен в allowlist (тот же gate). Убирает ручной env-токен.
   - `/b24/bot/event` валиден, если токен **захвачен** (при установке ИЛИ из события бота) ИЛИ совпал с env `B24_BOT_APPLICATION_TOKEN` (последнее средство). Покрыто юнит/интеграционными тестами (app-store + эндпоинт).
   - ⚠️ Портал-QA: фактическая форма `auth`-блока в `ONAPPINSTALL` и что `app.info` реально подтверждает — сверяется только на живом портале.

2. ✅ **Проводка установки (сделано).**
   - ✅ scope **`imbot`** в `getRequiredRights()`.
   - ✅ **клиентская регистрация** (`ui/app/utils/register-bot.ts`, зовётся из `useInstall.ts`):
     **LEGACY** `imbot.register` + `imbot.command.register` через SDK-действие
     `frame.actions.v2.call.make` (НЕ устаревший `BX24.callMethod`), best-effort, с юнит-тестами.
     _(Старый портал заказчика отдаёт 404 на `imbot.v2.Bot.register`; v2-код закомментирован — вернём при тираже, #243.)_
   - ⏳ **карточка приложения (портал):** scope `imbot` + «Ссылка-callback для события установки» = `…/b24/app/event`; **переустановить** (без этого ни регистрация бота, ни приём `ONAPPINSTALL` не сработают).

3. **Ручная проверка на ТЕСТОВОМ Битрикс24** (это и есть «оставить только ручную проверку») — сценарии в [docs/B24_BOT.md §12](B24_BOT.md): бросить счёт → получить сделку + 👍/👎; несколько файлов; сообщение без файла; кривой документ; клик 👍/👎 → метрика растёт; неверный токен → 403. ## 7. Заметка на будущее: обновления зависимостей

В окружении (на раннере CI **и** локально) включена политика pnpm **`minimumReleaseAge`** — она **отклоняет** пакеты, опубликованные меньше ~суток назад (защита от свежих скомпрометированных версий supply-chain). Важно: это **глобальный конфиг pnpm в окружении, его НЕТ в репозитории** (не ищите `.npmrc`/`pnpm-workspace.yaml` в дереве — там его нет). Что это значит на практике:

- если `pnpm up` подтянул совсем свежую версию — CI её **завернёт** на шаге `pnpm install --frozen-lockfile` с ошибкой `ERR_PNPM_MINIMUM_RELEASE_AGE_VIOLATION` (так и было с `uuid@14.0.1` в PR #223);
- решение: **подождать**, пока версия «отлежится» (по умолчанию — так, не обходим политику ради рутинного апдейта). Если версия нужна срочно — pnpm при `pnpm up` сам предлагает добавить её в `minimumReleaseAgeExclude` (пишет per-package `pnpm-workspace.yaml`); это **осознанный обход** для конкретного пакета, не для рутины.
