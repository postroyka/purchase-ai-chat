# Подготовка сервера procure-ai

Документ для тех, кто разворачивает и проверяет приложение на сервере.
Простыми словами: что это, что уже сделано, как проверить и что дальше.

---

## 1. О чём работа (коротко)

`procure-ai` — приложение, которое из загруженного прайс-листа поставщика
(PDF, XLSX/XLS, DOCX, JPG/PNG) автоматически создаёт сделку в Bitrix24 (воронка «Закупки»).
Текст из файла извлекается на сервере (PDF/изображения — OCR, office — python-хелпер), структуру достаёт агент.

Приложение работает в Docker и состоит из нескольких контейнеров:

| Контейнер | Что делает |
|---|---|
| `procure-app` | Сайт + бэкенд: приём файлов, обработка, отчёт |
| `procure-mcp` | Прослойка к Bitrix24 (наружу не торчит) |
| `procure-redis` | Хранилище заданий |
| `procure-watchtower` | Сам обновляет приложение при выходе новой версии |
| `nginx-proxy` + `acme-companion` | Внешний вход + HTTPS-сертификат |

Образы собираются автоматически на GitHub и скачиваются на сервер из
приватного хранилища GHCR. Исходный код на сервер **не** клонируется.

---

## 2. Что сделано на сервере (отчёт)

Сервер — Ubuntu 24.04. Выполнено:

1. Установлены `curl`, `make`, Docker (с правом запуска без `sudo`).
2. Открыты порты 80 и 443, настроен firewall. Исходящий HTTPS к `api.nbrb.by` не заблокирован (курс USD→BYN для метрик; при блокировке — тихий фолбэк на `USD_BYN_RATE`). Бэкенду также нужен исходящий HTTPS к порталу Bitrix24: `POST /session/b24` вызывает `app.info` на портале для проверки сессии из фрейма. Для self-hosted коробки убедитесь, что сервер достаёт портал по HTTPS.
3. Скачаны конфиги проекта (`docker-compose.*`, `Makefile`, `.env.prod.example`).
4. Заполнен `.env.prod` (секреты, домен). **Остаётся вписать рабочий вебхук Bitrix24.**
5. Выполнен вход в приватное хранилище образов: `docker login ghcr.io` → `Login Succeeded`.
6. Создана сеть `proxy-net`, запущен `nginx-proxy` + `acme-companion`.
7. Запущен основной стек (`make prod-up`) — все контейнеры поднялись и здоровы.

**Открытый вопрос:** HTTPS-сертификат Let's Encrypt на момент написания ещё
выпускался (acme-companion повторяет попытки). Это нужно подтвердить проверкой
из раздела 3 — сайт по `https://` должен открываться.

## 3. Чек-лист ручной проверки

Отмечайте по порядку. Команды для проверки — в разделе 4 (запускаются скриптом).

- [ ] **Контейнеры запущены.** `docker ps` показывает 6 контейнеров со статусом `Up`
      (`procure-app`, `procure-mcp`, `procure-redis`, `procure-watchtower`,
      `nginx-proxy`, `acme-companion`).
- [ ] **Бэкенд жив.** `app /health` отвечает `{"ok":true,"redis":"ok"}`.
- [ ] **MCP жив.** `mcp /api/health` отвечает `{"status":"ok",...}` (поле `service` убрано в шаблоне v0.3.0 — проверяем только живость).
- [ ] **Redis жив.** `redis-cli ping` отвечает `PONG`.
- [ ] **Сертификат выпущен.** В логах `acme-companion` есть «Certificate deployed»,
      сертификат имеет срок действия.
- [ ] **Сайт открывается снаружи.** `https://purchase.postroyka.by` отвечает `200`
      (UI отдаётся открыто — HTTP Basic убран, чтобы фрейм Битрикс24 мог загрузить страницу;
      браузерного запроса пароля больше нет).
- [ ] **Вход на страницу работает.** Открытая отдельно в браузере (вне портала) страница
      показывает встроенную форму входа и пускает по логину/паролю из `.env.prod`. Внутри
      Битрикс24 форма не появляется — сессия ставится автоматически через фрейм.
- [ ] **SESSION_SECRET** (при нескольких инстансах за балансировщиком). Если запускается более одного контейнера app, `SESSION_SECRET` должен быть задан явно и одинаково на всех — иначе cookie, выданная одним инстансом, не пройдёт проверку на другом.
  > ⚠️ Смена `BACKEND_API_TOKEN` или `PUBLIC_PAGE_BASIC_AUTH_PASS` инвалидирует все активные сессии, если `SESSION_SECRET` не задан явно (ключ cookie выводится из пароля/токена). Задайте `SESSION_SECRET` явно в проде, чтобы ротация учётных данных не выбивала пользователей.
- [ ] **(после вебхука Б24)** Загрузка тестового файла создаёт сделку в Bitrix24.

> Пункт «создаёт сделку» требует настроенного вебхука Б24 и выложенных
> REST-контроллеров (`make deploy-b24`). При корректной конфигурации загрузка
> тестового файла создаёт реальную сделку; ошибка на этом шаге — повод проверить
> вебхук/деплой контроллеров, а не ожидаемое состояние.
>
> Порядок безопасной выкатки PHP (staging `tstb24` → prod `b24`), бэкап и откат —
> см. `b24-controller/README.md` (`make deploy-b24` — dry-run по умолчанию,
> `APPLY=1` — реальная выкладка с бэкапом и health-чеком).

## 4. Скрипты проверки (запустить и прислать вывод)

Скрипты ничего не меняют — только читают состояние и печатают отчёт.
Цель: не набирать команды руками, а запустить одну строку и прислать результат.

> **Сначала доставьте скрипт на сервер.** Репозиторий на сервер не клонируется —
> в `~/procure-ai/` лежат только compose-файлы и `Makefile`. Поэтому скопируйте
> туда `scripts/smoke-test.sh` одним из способов:
>
> ```bash
> # Вариант 1 — с локальной машины (где есть доступ к репозиторию) через scp:
> scp scripts/smoke-test.sh ubuntu@СЕРВЕР:~/procure-ai/
>
> # Вариант 2 — прямо на сервере, из приватного репо через GitHub API (нужен PAT с Contents:Read):
> cd ~/procure-ai
> export GH_PAT="ghp_ваш_токен"
> curl -fsSL -H "Authorization: Bearer $GH_PAT" -H "Accept: application/vnd.github.raw" \
>   -o smoke-test.sh \
>   "https://api.github.com/repos/postroyka/purchase-ai-chat/contents/scripts/smoke-test.sh?ref=main"
> unset GH_PAT   # убрать токен из окружения сразу после использования
> ```

### На сервере (Linux/Ubuntu)

```bash
cd ~/procure-ai
bash smoke-test.sh
```

Файл: `scripts/smoke-test.sh`. Проверяет контейнеры, внутренние health-проверки,
сертификат, HTTPS и авторизацию (включая позитивную проверку токена — берётся из
`.env.prod` рядом со скриптом). В конце — строка ИТОГ (сколько OK / FAIL).

**Сквозная проверка (создаёт задание).** `scripts/e2e-upload.sh` загружает тестовый
PDF и опрашивает статус до завершения — прогоняет весь путь nginx → backend → агент → MCP
(доставляется на сервер так же, как `smoke-test.sh`):

```bash
cd ~/procure-ai && bash e2e-upload.sh
```

При настроенном вебхуке Б24 и выложенных контроллерах ожидается `status=done` (сделка
создана). `status=error` означает реальную проблему (вебхук/деплой/данные счёта), а не
норму — стоит посмотреть `processingLog`. Windows-вариант: `e2e-upload.ps1`
(PowerShell 7+, запускать с `-Token "<BACKEND_API_TOKEN>"`).

> **Важно про NAT.** Сервер за NAT не может достучаться до себя по публичному
> домену. Поэтому скрипт по умолчанию проверяет HTTPS/сертификат через локальный
> nginx-proxy приёмом `curl --resolve <домен>:443:127.0.0.1` — соединение идёт на
> `127.0.0.1`, но имя домена и сертификат проверяются как настоящие. Чтобы
> проверить через обычный DNS (например, с внешней машины), запустите:
> `RESOLVE_IP="" bash smoke-test.sh`. Для самоподписанного сертификата —
> `INSECURE=1 bash smoke-test.sh`.

### С обычного компьютера (Windows)

```powershell
powershell -ExecutionPolicy Bypass -File .\smoke-test.ps1
```

Файл: `scripts/smoke-test.ps1`. Проверяет снаружи: DNS, порт 443, сертификат,
`/health`, главную страницу.

> Как получить скрипт на сервер/компьютер: скопировать файл из папки `scripts/`
> репозитория (или скачать его содержимое и сохранить под тем же именем).

### Сквозной тест агента (#36 runAgent)

`smoke-test.sh` проверяет инфраструктуру, но не запускает агента. Чтобы
проверить весь путь «файл → агент → MCP», есть `scripts/agent-e2e-test.sh`:
он грузит тестовый счёт через `/upload`, опрашивает `/job/:id/status`,
достаёт логи агента из контейнера `procure-app` и выносит понятный вердикт.

Доставьте на сервер **скрипт и эталонный счёт** (рядом друг с другом):

```bash
# через scp с машины, где есть репозиторий (обе цели одной командой):
scp scripts/agent-e2e-test.sh scripts/samples/etalon-invoice.pdf ubuntu@СЕРВЕР:~/procure-ai/

# затем на сервере:
cd ~/procure-ai
grep -q '^ANTHROPIC_API_KEY=..' .env.prod || echo "⚠ ANTHROPIC_API_KEY не задан в .env.prod — агент упадёт!"
bash agent-e2e-test.sh
```

Скрипт сам подхватит лежащий рядом `etalon-invoice.pdf`. Свой файл —
`FILE=/path/to/price.xlsx bash agent-e2e-test.sh`.

> **Локально на Windows** (где claude авторизован по подписке, вне Docker) —
> та же проверка одной командой через PowerShell-версию:
> `powershell -ExecutionPolicy Bypass -File .\scripts\agent-e2e-test.ps1`
> (по умолчанию `http://localhost:3000` и эталон из `scripts/samples/`).

> **⚠️ Авторизация claude в Docker.** Подписочная сессия Claude.ai внутри
> контейнера не работает — агенту нужен `ANTHROPIC_API_KEY` в `.env.prod`
> (он пробрасывается в контейнер app). Без него агент вернёт «Not logged in»
> ещё до обращения к MCP.

> **♻️ Ретрай агента (#104).** Транзиентные сбои провайдера (HTTP 429/5xx,
> сеть, таймаут) агент повторяет с бэкоффом: `AGENT_MAX_ATTEMPTS` (по умолч. 3),
> `AGENT_RETRY_BASE_MS`, `AGENT_RETRY_MAX_MS` (подробности — в `.env.prod.example`).
> `AGENT_MAX_ATTEMPTS=1` выключает ретрай. ⚠️ У каждой попытки полный
> `AGENT_TIMEOUT_MS`, поэтому при недоступности провайдера задание может держать
> слот до `AGENT_MAX_ATTEMPTS × AGENT_TIMEOUT_MS`; с `MAX_CONCURRENT_JOBS=2`
> новые загрузки в это время получают `429`. Если важнее «быстро падать» —
> снизьте `AGENT_MAX_ATTEMPTS` или `AGENT_TIMEOUT_MS`.

> **Что считать успехом.** Полный флоу: агент запустился, авторизовался, прочитал
> файл, вызвал инструменты `b24_pst_crm_*` и **создал сделку** в Bitrix24
> (`status=done`). Если вебхук Б24 / контроллеры ещё не настроены, агент дойдёт до
> MCP и упрётся в недоступность инструментов — это конфигурационное состояние, а не
> штатный результат.

## 5. Два compose-стека и изоляция проектов (важно!)

Приложение и реверс-прокси — это **два отдельных compose-стека** с **разными именами проектов**:

| Стек | Файл | Проект | Команда |
|---|---|---|---|
| Приложение (app, mcp, redis, watchtower) | `docker-compose.prod.yml` | `procure-ai` | `make prod-up` / `prod-redeploy` |
| Реверс-прокси (nginx-proxy, acme-companion) | `docker-compose.nginxproxy.yml` | `procure-proxy` | `make init-nginxproxy` |

> **TRUST_PROXY.** За nginx-proxy дефолт (`1`) корректен: бэкенд читает реальный IP клиента из `X-Forwarded-For`, и per-IP лимиты `/login` и `/session/b24` (защита от брутфорса) работают корректно. При нестандартной топологии (несколько прокси-слоёв, CDN) задайте `TRUST_PROXY` явно: число хопов, `loopback` или список IP/CIDR. Без корректного значения все клиенты видятся с IP прокси и подпадают под один общий лимит.

> ⚠️ **Лимит размера загрузки на nginx-proxy.** У nginx-proxy дефолтный `client_max_body_size`
> — **1 МБ**, а приложение разрешает файлы до 20 МБ (`MAX_FILE_SIZE_MB`). Без правки большой
> файл упрётся в `413 Request Entity Too Large` ещё на прокси (до backend он не дойдёт). Поднимите
> лимит для вашего хоста, создав файл `vhost.d/<VIRTUAL_HOST>` (имя = значение `VIRTUAL_HOST` из
> `.env.prod`) рядом с конфигом nginx-proxy с одной строкой:
> ```nginx
> client_max_body_size 25m;
> ```
> (25 МБ — с запасом к 20 МБ; согласуется с `NUXT_MAX_ATTACH_MB`.) Затем перезапустите nginx-proxy.

> 🛠️ **Деплой без GitHub Actions.** Обычно прод-образ (`latest`) публикует workflow
> `Publish images` на каждый зелёный push в `main` (continuous deploy; тег `v*` —
> необязателен, лишь добавляет версию-метку), и его подхватывает Watchtower. Если
> раннеры/минуты Actions недоступны —
> собрать и запушить образы вручную с любой машины с Docker:
> `GHCR_TOKEN=ghp_xxx make deploy-images` (PAT со scope `write:packages`), затем на
> сервере `make prod-redeploy`.

> 🩹 **Грабля (исправлена).** Раньше у обоих файлов не было поля `name:`, и Docker
> присваивал им **одно** имя проекта (по имени каталога). Из-за этого
> `make prod-redeploy` (внутри — `up --remove-orphans`) удалял `nginx-proxy` и
> `acme-companion` как «orphans» — на проде падал весь HTTPS. Теперь имена разные
> (`procure-ai` / `procure-proxy`), и `--remove-orphans` затрагивает только свой стек.

**TLS-сертификат хранится в томах `procure-ai_nginx-certs` и `procure-ai_acme-data`.**
Проверить и не удалить по ошибке:
```bash
docker volume ls | grep -E 'nginx-certs|acme-data'   # ожидаем procure-ai_*
```
> ⚠️ Эти тома объявлены в стеке прокси (`procure-proxy`), поэтому опасны именно
> `docker compose -p procure-proxy down -v` или `docker volume rm procure-ai_nginx-certs
> procure-ai_acme-data` — они удалят сертификат (будет перевыпущен при следующем
> `init-nginxproxy`). Обычные `down`/`down -v` стека приложения (`procure-ai`) их **не трогают**.

### Разовая миграция уже работающего сервера на изоляцию проектов
Если прокси сейчас поднят под старым (общим) проектом, перевести его на `procure-proxy`:

> ⚠️ **Кратковременный downtime.** Между `docker rm -f` и стартом нового прокси порты
> 80/443 не отвечают (обычно &lt;1 мин). Выполняйте в нерабочее время. Порядок важен:
> если запустить `init-nginxproxy` до `rm -f`, будет ошибка `container name already in use`.

```bash
cd ~/procure-ai
docker rm -f nginx-proxy acme-companion   # снести старые контейнеры (тома с cert остаются)
make init-nginxproxy                       # поднять заново уже в проекте procure-proxy (те же тома)
make prod-redeploy                         # теперь безопасно
bash smoke-test.sh
```
> Если `init-nginxproxy` упал с `name already in use` — `docker rm -f` прошёл не до конца, повторите его.

## 6. Что дальше

1. Дождаться/подтвердить выпуск TLS-сертификата и зелёный `https://`
   (если долго `503` в challenge — перезапустить `docker restart acme-companion`).
2. Вписать рабочий вебхук Bitrix24 в `.env.prod` и выполнить `make prod-redeploy`.
3. Прогнать `smoke-test.sh` на сервере и `smoke-test.ps1` с рабочего компьютера,
   пройти чек-лист из раздела 3.

> **После деплоя: переход с HTTP Basic на app-сессию.** Браузер больше не показывает Basic-промпт.
> Standalone (вне портала): встроенная форма входа `/login` — проверьте вход по логину/паролю из `.env.prod`.
> Внутри Bitrix24: сессия ставится молча через `/session/b24` — промпта нет, страница должна открываться сразу.
