# Производительность разбора файлов («быстро/медленно»)

`Последняя ревизия: 2026-06-23`

Документ задаёт **ориентиры скорости** разбора одного файла: что считать быстрым, нормальным и
медленным. Пороги — **оценочные** (выставлены «на глаз» по наблюдениям недели 4); калибруйте реальностью
по данным `/metrics` и логу замеров (`SHOW_TIMINGS`, см. ниже), правьте env — код менять не нужно.

## Из чего складывается время файла

`durationMs` (полное время файла) ≈ **извлечение текста** + **работа агента** (+ возможный ретрай-бэкофф):

| Этап | Что это | Оценка | От чего зависит |
|---|---|---|---|
| **Извлечение** | `pdftotext` (цифровой PDF) / `office` (docx/xlsx) | ~0.1–3 с | размер файла |
| **Извлечение** | `ocr` (скан/фото, tesseract) | ~10–60 с | число страниц × ~2–5 с/стр, DPI |
| **Агент** (`agentMs`) | прогон Claude Code: распознавание позиций, матчинг поставщика/договора/артикулов, создание сделки | ~30–75 с | число позиций, число обращений к MCP/B24 |
| Ретрай | бэкофф при транзиентном сбое провайдера (429/5xx) | 0, либо +сек | редко |

**Вывод:** обычно доминирует **агент**, а самый частый «виновник медленного файла» — **OCR** (скан вместо
цифрового PDF). Поэтому в логе замеров показываем метод **и точное время** извлечения
(`извлечение: ocr 2.3 с`, #203.2 — мерится отдельно вокруг extractFn в `agent-runner.js`) и время агента.

**«Думает vs ищет» (#222):** у агента есть **число ходов** — по файлу в логе (`агент 44.2 с (12 ходов)`) и
**среднее** на `/metrics` («Среднее время агента» → `~8 ходов/прогон`). Дешёвый сигнал, куда уходит время:
**много ходов** = агент много раз ходил в инструменты (поиск/итерации, «ищет»); **мало ходов при большом
времени** = долго «думал» на ход. Грубое правило: `agentMs / ходы` — большое → думает/медленные инструменты,
много ходов → ищет. ⚠️ `agentMs` включает и ожидание инструментов (MCP/Б24), поэтому время‑на‑ход — лишь
ориентир; точные тайминги по КАЖДОМУ REST‑вызову к Bitrix24 логирует MCP — см. «REST-тайминги к Bitrix24
(#262)» ниже.

## REST-тайминги к Bitrix24 (#262)

`agentMs` включает и **сетевые REST-запросы к Bitrix24** (поиск поставщика/договора/товара,
создание сделки) — а портал/сеть это частый подозреваемый на «медленно». Чтобы отделить «думание»
модели от ожидания портала, MCP **логирует длительность каждого REST-вызова** вокруг `callV2`
(обёртка `timedCallV2` в `mcp-overlay/server/utils/rest-timing.ts`).

Формат строки в логах MCP-контейнера (грепается по `[rest-timing]`):

```
[rest-timing] method=procuresupplier.findbyunp ms=940 ok=true
[rest-timing] method=procurecontract.find ms=2514 ok=true
[rest-timing] method=procureproduct.findbyvendorcode ms=315 ok=true
[rest-timing] method=procuredeal.create ms=1820 ok=true
```

- `ms` — **wall-time** round-trip (включая TLS-прогрев на первом вызове; именно его «видит» агент).
- `ok` — успех/ошибка REST-вызова (ошибка тоже логируется, затем пробрасывается).
- Уровень — `notice`: виден при дефолтном уровне логов (INFO и выше), без `NUXT_LOG_LEVEL=debug`.

Покрыт одиночный `callV2` (через него ходят все 4 инструмента). Пакетные `batchV2`/`batchV3`
ещё не используются и пока не инструментированы — когда появятся (задел под батчинг поиска
товаров), к ним добавится отдельная обёртка `timedBatchV2`.

**Смотреть:** `docker compose -f docker-compose.prod.yml logs mcp 2>&1 | grep rest-timing`.

### Сколько внутри agentMs ушло на REST к порталу (агрегат по логам)

⚠️ REST-время — это **часть** `agentMs`, а не параллельная ему величина (`agentMs` включает ожидание
инструментов, см. выше). Так что речь не о «REST или агент», а о том, **какая доля `agentMs` ушла на
round-trip'ы к Bitrix24**.

Строки `[rest-timing]` машиночитаемы (`key=value`), поэтому суммарное REST-время за **временно́е окно**
считается прямо из логов MCP-контейнера — grep + awk, без доработок кода:

```bash
docker compose -f docker-compose.prod.yml logs --since 1h mcp 2>&1 \
  | grep -o '\[rest-timing\].*' \
  | sed -E 's/.*method=([^ ]+) ms=([0-9]+) ok=([a-z]+).*/\1 \2 \3/' \
  | awk '{n[$1]++; s[$1]+=$2; if($2>m[$1])m[$1]=$2; if($3=="false")e[$1]++; S+=$2; N++}
         END{for(k in n) printf "%-32s n=%d sum=%dms avg=%dms max=%dms err=%d\n",k,n[k],s[k],s[k]/n[k],m[k],e[k];
             printf "%-32s n=%d sum=%dms (=%.1fs)\n","ВСЕГО REST",N,S,S/1000}'
```

Даёт по каждому методу число вызовов / сумму / среднее / максимум / число ошибок и **«ВСЕГО REST»** за окно.

Реальная строка в `docker logs` богаче примера выше (формат логгера + имя контейнера) — `grep -o` и `sed`
это срезают, парсинг не ломается:
```
procure-mcp  | NOTICE [bx24-template-mcp] NOTICE: [rest-timing] method=procuredeal.create ms=1820 ok=true {} {} 2026-06-23 10:00:00
```

**Оговорки:**
- Это **агрегат за окно `--since`**, НЕ по конкретной загрузке: в строке нет job-id, поэтому привязать REST
  к одному файлу/сделке из логов нельзя. Для грубой картины «сколько в среднем уходит на портал» — годится;
  для per-file разбивки рядом с `agentMs` нужен Шаг 2 (ниже).
- Работает при `NUXT_LOG_LEVEL` ≤ `notice` (по умолчанию `info` — ок; при `warning`+ строки `[rest-timing]`
  в лог не попадут).

> **Почему это пока агрегат по логам, а не строка `restMs` в «Логе обработки» (in-app, Шаг 2, #262).**
> Per-file `restMs` рядом с `agentMs`/`extractMs` требует протащить сумму из **MCP-контейнера** в
> **backend-процесс**: REST-вызовы и их тайминги живут в MCP, а «Лог обработки» собирает backend
> (`agent-runner.js`, `onMeta`). Прямого канала нет — агент запускается с `--output-format json`, поэтому
> backend видит только финальный агрегат, а не промежуточные ответы инструментов; Redis к MCP не подключён.
> Варианты (парсинг `stream-json` в backend **или** Redis-side-channel с job-id в заголовке `--mcp-config`)
> — отдельная инфраструктурная работа; зафиксирована как Шаг 2 в [ROADMAP §4.3](ROADMAP.md).

**Что показали замеры (тестовый портал):** серверная обработка самого Bitrix — **десятки мс**
(поле `time.duration` в сыром REST-ответе ~0.03–0.11 с), а wall-time доминируют **сетевой
round-trip и холодный TLS-хендшейк** (первый вызов ~1–2.5 с, последующие 0.3–0.9 с по тёплому
соединению). Вывод: «медленно» делает **не вычисление портала, а число и латентность сетевых
round-trip'ов**; кандидат на оптимизацию — батчинг поиска товаров (тема для ROADMAP).

> Серверную `duration` Bitrix (`time` в ответе) `callV2` отбрасывает на вендоренной границе
> (`mcp/server/utils/sdk-helpers.ts`) — её видно при ручной диагностике сырым REST-вызовом
> (`curl … -w '%{time_total}'`), не в `[rest-timing]`.

## Пороги по полному времени файла (мои оценки)

| Класс | Полное время `durationMs` | Трактовка |
|---|---|---|
| 🟢 **быстро** | **≤ 45 с** | цифровой счёт, немного позиций, быстрый агент |
| ⚪ **норма** | > 45 и < 90 с | типичный файл (на проде среднее время агента ≈ 75 с) |
| 🟡 **медленно** | **≥ 90 с** | OCR-скан, много позиций, повторы матчинга или ретрай |

Базис оценки: на проде недели 4 «Среднее время агента ≈ 74.8 с» → типичный файл ~75–85 с (это «норма»);
заметно выше (OCR + матчинг) → «медленно»; явно ниже (цифровой, мало позиций) → «быстро».

⚠️ Пороги по умолчанию рассчитаны на **цифровые PDF** (быстрое извлечение). У типичного **скана** OCR
добавляет ~10–60 с поверх агента, поэтому «обычный» скан легко уходит за 90 с и красится в «медленно».
Если поток в основном сканы — поднимите сначала `TIMING_SLOW_MS` (напр. до 120000–140000).

> Это **стартовые** числа. Соберите 20–50 реальных файлов с `SHOW_TIMINGS=true`, посмотрите распределение
> и сдвиньте пороги: «быстро» ≈ 25-й перцентиль, «медленно» ≈ 75–90-й. Правится без передеплоя кода.

## Настройка (env)

| Переменная | Контейнер | По умолчанию | Смысл |
|---|---|---|---|
| `TIMING_FAST_MS` | app | `45000` | `durationMs` ≤ этого → «быстро» |
| `TIMING_SLOW_MS` | app | `90000` | `durationMs` ≥ этого → «медленно»; между — «норма» |
| `SHOW_TIMINGS` | app | `false` | **детальная** строка замеров в лог («⏱ всего … — <класс> · агент … · извлечение: …»). Живой mm:ss «обрабатывается N сек» — **всегда**, без флага (#203) |

Классификацию считает бэкенд (`classifySpeed` в `backend/index.js`) и отдаёт полем `speed`
(`fast|normal|slow`) в `GET /job/:id/status` — **только** при `SHOW_TIMINGS=true`. UI рисует класс в логе и
подсвечивает «медленно» янтарным. **Пер-файловый** `speed` в метрики не пишется (это лог на странице), но
**агрегированное распределение** «быстро/норма/медленно» по **успешно разобранным** файлам (status done;
ошибки/таймауты не входят) пишется в метрики и показывается на `/metrics` плиткой «Скорость разбора»
(issue #207 — агрегат, не сырые тайминги).

## Где смотреть факт

- **Лог на странице** (`SHOW_TIMINGS=true`): по каждому файлу «⏱ всего 1 мин 20 с — норма · агент 1 мин 12 с
  (12 ходов) · извлечение: pdftotext 0.4 с» (метод+время #203.2, число ходов агента #222).
- **`/metrics`** → «Среднее время агента» (lifetime) — ориентир для калибровки порога «норма»; плитка
  **«Скорость разбора»** — распределение быстро/норма/медленно за всё время (#207).

## Как калибровать (коротко)

1. Включить `SHOW_TIMINGS=true` на staging/проде на время замеров.
2. Прогнать репрезентативную пачку (цифровые + сканы, мало/много позиций).
3. Записать `durationMs` (видно в логе/Network `…/status`).
4. Сдвинуть `TIMING_FAST_MS` / `TIMING_SLOW_MS` под реальное распределение, перезапустить app-контейнер.
