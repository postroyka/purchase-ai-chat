# ТЗ: интеграция с «1С: Управление торговлей» через MCP

`Последняя ревизия: 2026-06-20` · Статус: черновик для команды (контракт сверен с PHP-контроллерами Bitrix24)

## 1. Контекст и две части работы

Система `procure-ai` принимает счёт поставщика (PDF/скан/Excel), ИИ-агент (Claude) распознаёт его и
**создаёт документ закупки** в учётной системе. Агент общается с учётной системой **только** через 4
инструмента MCP-сервера. Сегодня эти инструменты ходят в Bitrix24 REST (`shef:purchase.api.*`). Задача —
сделать так, чтобы те же 4 операции выполнялись в **1С: УТ**.

Это **две независимые части** (не «перенацелить одним флагом» — такого флага сейчас нет):

1. **Сторона 1С (этот документ, §2–§3, §8).** 1С выставляет HTTP-сервис (REST/JSON) с 4 методами по
   контракту ниже. По нему 1С-программист может работать **автономно**.
2. **Сторона MCP (§4–§5).** В нашем коде каждый инструмент (`mcp-overlay/server/mcp/tools/deals/*.ts`)
   зовёт `useBitrix24Tenant()` + `callV2(b24, 'shef:purchase.api...')` — это **жёстко завязано на Bitrix
   SDK** (`mcp/server/utils/sdk-helpers.ts`, `bitrix24-tenant.ts`): `B24Hook.fromWebhookUrl`, `TypeB24`,
   разбор `AjaxResult`. `callV2` возвращает **конверт Bitrix REST**, а не «плоский» JSON. Чтобы ходить в
   1С, на MCP-стороне нужен **новый путь вызова** (хелпер `call1C()` или адаптер с тем же интерфейсом) и
   **переключатель бэкенда на установку** (по образцу существующего `NUXT_BITRIX24_OAUTH_ENABLED` —
   напр. `NUXT_BACKEND=b24|1c` в `useBitrix24Tenant`). Логика самих инструментов и агента не меняется,
   меняется слой транспорта.

```
[Агент Claude] → [MCP-инструмент] → callV2 → Bitrix24 REST shef:purchase.api.*   (сейчас)
                                  → call1C → HTTP-сервис 1С: УТ                  (цель, §2)
```

**Файлы (вложение) не идут через 1С-волюм.** Вход агент→MCP остаётся `filePath`; MCP-сервер сам читает
файл из read-only тома `uploads` и **base64-кодирует** его, после чего шлёт в `createDeal` уже `fileName` +
`fileContent` (base64). 1С получает только base64 и **никогда** не обращается к тому `uploads` — этот кусок
для 1С не меняется.

## 2. Контракт операций (4 метода)

JSON, UTF-8, числа с точкой-разделителем. **«Не найдено» — это не ошибка**: возвращайте `{ "id": null }`.
Реальная ошибка (нет связи/исключение) — тело `{ "error": true, "code": "...", "message": "..." }`.

### 2.1. Найти поставщика по УНП — `findSupplierByUnp`

| | |
|---|---|
| Вход | `{ "unp": "123456789" }` |
| Выход (найдено) | `{ "id": 4021, "title": "ООО Ромашка", "unp": "123456789" }` |
| Выход (не найдено / невалидный УНП) | `{ "id": null }` |

- УНП нормализуется (убрать пробелы, NBSP, дефисы). После нормализации — **ровно 9 цифр**; иначе
  (невалидный формат) → `{ "id": null }` (не ошибка). Слишком длинный (> 32 симв.) → ошибка `sup:011`.
- Матчинг: реквизит **ИНН/УНП** контрагента, **страна = Беларусь** (в B24 `COUNTRY_ID=4`), **точное** «=».
- Несколько контрагентов с одним УНП → **минимальный id**.
- **Вне scope:** российские поставщики (ИНН+КПП без УНП).

### 2.2. Найти договор поставщика — `findContract`

| | |
|---|---|
| Вход | `{ "supplierId": "4021", "number": "243Э20", "date": "15.03.2025" }` — `number`/`date` опциональны |
| Выход (найдено) | `{ "id": 88, "number": "243Э20", "date": "15.03.2025" }` |
| Выход (не найдено) | `{ "id": null }` |

Отбор кандидатов (в B24 — SQL-фильтр; для 1С — эквивалентные условия):
- привязан к поставщику `supplierId`;
- **активен** (флаг активности);
- **вид договора** ∈ {закупка, закупка-комиссионный} (в B24 — коды `PURCHASE`, `PURCHASE_ZAK`);
- **не удалён / не «Брак»** (в B24 — STATUS ≠ TO_DELETE).

Затем по `number`/`date` сверяем **в приложении** (не в SQL), среди кандидатов поставщика:
- **`number`** — гомоглиф-устойчиво + срез хвостовой пометки в скобках через пробел. Алгоритм
  `matchableNumber` (см. §3.1): `foldHomoglyphs(trim(strip_trailing_paren(n)))`, сравнение `===`.
  Примеры: `789-22/24 (основной)` ≡ `789-22/24`; `243Э20` (лат. E) ≡ `243Э20` (кир. Э); но `ТМ-100(А)`
  (без пробела) — скобка **не** срезается. Если `number` не передан — по номеру не фильтруем.
- **`date`** — сравниваем **отображаемое** значение в формате `d.m.Y`, точно (trim). Не передан — не
  фильтруем.
- Несколько совпадений → **минимальный id** (перебор по возрастанию id, первый подошедший).

> Договор **обязателен** для документа закупки: если агент не нашёл договор (`id:null`), он
> останавливается и документ **не создаёт**. Поэтому корректный матчинг договоров критичен.

### 2.3. Найти номенклатуру по артикулу — `findProductByVendorCode`

| | |
|---|---|
| Вход | `{ "vendorCode": "тех 100х25х6000" }` |
| Выход (найдено) | `{ "id": 9100, "name": "Доска 100×25×6000", "vendorCode": "тех 100х25х6000" }` |
| Выход (не найдено) | `{ "id": null }` |

- Матчинг — **строго ТОЧНОЕ совпадение артикула «как есть»** (один точечный `=` по полю «Артикул»
  активной «родительской» номенклатуры). **Гомоглиф-фолдинг и LIKE-поиск намеренно убраны** («выбрал →
  поискал → не нашёл → записал в лог»). Это **отличие от договоров** (там гомоглифы есть).
- Несколько совпадений → **минимальный id**. Длина артикула > 64 → ошибка `prd:011`.
- **Только по артикулу.** Поиск по наименованию намеренно НЕ поддерживается (в API всегда идёт артикул; агент может лишь ИЗВЛЕЧЬ артикул из текста наименования по пометке «арт.», но в запрос уходит код, не наименование).
- Нет артикула в каталоге → `{ "id": null }` (это нормально; см. правило позиций в §2.4).

### 2.4. Создать документ закупки — `createDeal`

Тело запроса:

```jsonc
{
  "supplierId": "4021",                 // контрагент-поставщик (из 2.1) — ОБЯЗАТЕЛЕН
  "responsibleUserId": "20",            // ответственный (менеджер), id пользователя 1С — ОБЯЗАТЕЛЕН
  "contractId": "88",                   // договор (из 2.2). На стороне агента обязателен; сервис
                                        //   должен принимать и его отсутствие (0 = не передан)
  "fileName": "schet_243.pdf",          // имя исходного файла (для вложения)
  "fileContent": "<base64>",            // содержимое файла, base64 (≤ ~34 МБ base64 ≈ 25 МБ бинарных)
  "documentDate": "15.03.2025",         // дата документа (d.m.Y), опц.
  "processingLog": "Распознано 7 позиций…", // лог обработки → в примечание/комментарий документа
  "items": [
    {
      "productId": "9100",              // номенклатура (из 2.3), опц.
      "vendorCode": "тех 100х25х6000",  // артикул из документа, опц.
      "name": "Доска 100×25×6000",      // наименование из документа
      "priceExclVat": 12.99,            // цена за единицу БЕЗ НДС (2 знака)
      "quantity": 224.8                 // количество, дробное до 2 знаков (≤0 → 1); единица всегда «шт»
    }
  ]
}
```

Выход:

```jsonc
{ "dealId": 31840, "warnings": ["no_items_matched"] }   // warnings — опционально
{ "dealId": null }                                       // при ошибке создания документа
```

**Обязательные поля и ошибки:**
- `supplierId` ≥ 1 и `responsibleUserId` ≥ 1 — иначе ошибка `deal:010`. `responsibleUserId` идёт в
  ответственного и автора записи в ленте.
- `items` **может быть пустым** (это не ошибка). Больше `MAX_ITEMS` (500) → ошибка `deal:021`.
  `fileContent` > 34 МБ base64 → `deal:022`.

**Правило позиций (важно, легко сделать наоборот):**
- В сделку кладём **только** позиции с сопоставленным товаром каталога (**`productId` непустой/&gt;0**).
- Позиция **без `productId`** → в документ **НЕ кладём** (пропускаем) — неважно, есть ли `vendorCode`:
  артикул не найден в каталоге **или** артикула нет вовсе. Такие позиции уходят в `processingLog`, а не
  в табличную часть (#258: свободные строки `PRODUCT_ID=0` больше не создаём).
- Фильтр на контроллере — `empty($item['productId'])` (ловит пусто/`'0'`/`null`/отсутствие); `productId`
  всегда положительный id товара, поэтому «0»/пусто = «не сопоставлено».

**НДС и суммы (бизнес-правило — НЕ «исправлять»):**
- В строку пишется **`PRICE = priceExclVat` КАК ЕСТЬ** (округлённая до 2 знаков, цена **нетто/без НДС**), плюс флаги
  **`TAX_RATE = 20`** и **`TAX_INCLUDED = N`** (#325) — цена нетто, учётная система **добавляет 20% сверху**.
  **НЕ домножайте цену на 1.20** сами — это сделает система. (Раньше при `=Y` система делила нетто на 1.2 и
  округляла → расхождение в 1 копейку; теперь цена вносится 1-в-1 со счётом.)
- `quantity` — дробное, до 2 знаков (#286; если не-финит/≤ 0 → 1). Единица — всегда «шт» (в B24 `MEASURE_CODE` ОКЕИ 796).

**Прочее:**
- `documentDate` (если валидная `d.m.Y`) → дата документа на **09:00**; нет/непарсибельна → текущая
  дата-время, и при «передана, но непарсибельна» добавляется `warnings: ["document_date_unparsed"]`.
- `fileContent` (base64) приложить к документу. Пустой — не прикладываем.
- `processingLog` — в примечание/комментарий документа (в B24 ещё и запись в ленту/таймлайн).
- В ответе вернуть **id созданного документа** как `dealId`; при сбое создания — `{ "dealId": null }`.

**Документ создаётся «лучшими усилиями», без отката:** сначала создаётся сам документ (если не вышло →
`{dealId:null}`), затем по отдельности — позиции, вложение, запись лога. Сбой любого из них **НЕ откатывает**
документ, а добавляет соответствующий `warning`. Возможные значения `warnings[]`:

| warning | когда |
|---|---|
| `no_items_matched` | ни одна позиция не легла в табличную часть (все либо без сопоставления, либо пусто) |
| `product_rows_failed` | позиции не сохранились |
| `file_attach_failed` | вложение не прикрепилось |
| `invalid_base64_file` | `fileContent` не декодировался |
| `document_date_unparsed` | `documentDate` передан, но не `d.m.Y`/невалидная дата |
| `timeline_comment_failed` | не удалось записать лог в ленту |

## 3. Сводка форматов («устаканились»)

| Поле | Формат | Примечание |
|---|---|---|
| УНП | 9 цифр | пробелы/NBSP/дефисы убрать; невалидный → `{id:null}` |
| Даты | `d.m.Y` | напр. `15.03.2025`; дата документа → 09:00 |
| Валюта | BYN | фиксировано |
| Цена | `priceExclVat` — за единицу, **без НДС**, 2 знака | в документе пишется как есть + `TAX_RATE=20`/`TAX_INCLUDED=N` (нетто, НДС сверху, #325); **не** ×1.20 |
| Количество | дробное, 2 знака (не-финит/≤0 → 1) | единица всегда «шт» |
| **Номер договора** | строка; **гомоглиф-устойчиво** + срез хвостового « (…)» | см. §3.1 |
| **Артикул** | строка; **строгое точное** совпадение | гомоглифов НЕТ (в отличие от договора) |
| Тай-брейк | **минимальный id** | при нескольких совпадениях |
| Файл | base64 | вложение; ≤ ~34 МБ base64 |
| id | **целое число** | во всех 4 методах |

### 3.1. Гомоглифы (только номер договора)

Оператор мог набрать номер договора в любой раскладке; в Unicode латиница/кириллица-двойники — разные
символы. Канонизация перед сравнением (`foldHomoglyphs`): **верхний регистр** + замена кириллических
двойников на латинские, затем `===`. Карта замены (кир.→лат.):

```
А→A В→B Е→E Ё→E К→K М→M Н→H О→O Р→P С→C Т→T У→Y Х→X І→I Ј→J Ѕ→S
```

`matchableNumber(n)` дополнительно срезает **хвостовую** аннотацию в скобках, **отделённую пробелом**
(`/\s+(\(...\))+$/`): `789-22/24 (основной)` → `789-22/24`. Скобку без пробела (`ТМ-100(А)`) не трогает.
Применяется **только** к номеру договора; артикул сверяется строго.

## 4. Маппинг сущностей и где «переключатель»

| Сущность | Bitrix24 (сейчас) | 1С: УТ (предложение — уточнить) |
|---|---|---|
| Поставщик | Компания, реквизит `RQ_INN`, страна BY | **Контрагент** (поиск по ИНН/УНП) |
| Договор | Список «Договора» (iblock 32), вид `PURCHASE`/`PURCHASE_ZAK` | **Договор с поставщиком** |
| Номенклатура | Каталог (iblock 15, по умолч.), свойство «Артикул» | **Номенклатура**, реквизит «Артикул» |
| Документ закупки | Сделка (воронка «Закупки» cat 1, стадия `C1:NEW`, BYN) | **Заказ поставщику** или **Поступление** (см. §7) |
| Ответственный | Пользователь B24 | **Пользователь/менеджер 1С** |

**Где в нашем коде делается переключение бэкенда:** слой `mcp/server/utils/` (`sdk-helpers.ts::callV2`,
`bitrix24-tenant.ts::useBitrix24Tenant`). Туда добавляется ветка выбора бэкенда (по образцу
`NUXT_BITRIX24_OAUTH_ENABLED`) и хелпер `call1C()`, реализующий те же 4 вызова против HTTP-сервиса 1С.
Инструменты в `mcp-overlay/.../deals/*.ts` остаются по контракту §2.

## 5. Транспорт и авторизация

- **HTTP-сервис 1С** (REST/JSON). 4 метода — отдельные URL либо один с полем `method`; согласуем при старте.
- Новый секрет на MCP-стороне: базовый URL 1С + токен (напр. `NUXT_1C_BASE_URL`, `NUXT_1C_TOKEN`). Должен
  **редактироваться в логах** (как сейчас вебхук-URL Bitrix, см. logger-redactor) — секрет не должен утечь.
- Таймауты/повторы: поиск — быстрый (десятки мс). Методы поиска — **идемпотентны**; `createDeal` — нет
  (создаёт). Повтор транзиентных сбоев — на стороне вызывающего.

## 6. Что вне scope

- Российские поставщики без УНП (ИНН+КПП).
- Поиск номенклатуры по наименованию (только по точному артикулу).
- Дедупликация документов закупки (создаём всегда).
- OAuth/мультиарендность (одна база 1С на инсталляцию).

## 7. Открытые вопросы к заказчику / 1С (решить до реализации)

1. **Какой документ 1С** = «сделка закупки»: **«Заказ поставщику»** (заказ/намерение, без движения
   товара/учётных проводок, проведение опционально) или **«Поступление товаров и услуг»** (приходует
   товар, может создавать долг перед поставщиком)? Источник — счёт поставщика со строками и ценами, что
   ближе к **Поступлению**, но у него тяжелее учётные последствия. Не «очевидно одно» — решает заказчик.
2. **«Активный договор» в терминах 1С:** какой реквизит = «активен», какой набор «вид договора» = закупка
   (вкл. комиссионную), как отсекать удалённые/брак.
3. Где в выбранном документе 1С хранить **вложение** (файл) и **лог обработки** (примечание/комментарий).
4. Маппинг `responsibleUserId` (id пользователя 1С) и прав на создание документа.
5. Создавать сразу **проведённым** или черновиком.
6. Подтвердить правило **«цена нетто/без НДС, система добавляет 20% сверху, без домножения вручную»** на стороне 1С (#325).

## 8. Приёмка

- 4 метода отвечают по контракту §2 (вкл. `{id:null}` на «не найдено», `{dealId:null}` на ошибку,
  `warnings[]`).
- Сквозной тест: счёт с известным поставщиком/договором/артикулами → документ с корректными позициями,
  ценами (НДС 20% «включён», без домножения), вложением и логом; `dealId` возвращается.
- Негатив: неизвестный/невалидный УНП → `{id:null}`; нет договора → агент останавливается (документ не
  создаётся); позиция без сопоставленного товара (`productId` пуст) — артикул не из каталога **или** нет
  артикула — → **отброшена** (в лог, не в документ; #258 — свободных строк больше нет); непарсибельная
  `documentDate` → документ + `warnings:[document_date_unparsed]`.
