# Обратная связь от сотрудника (`POST /feedback`)

`Последняя ревизия: 2026-06-19`

Это **первый из трёх каналов** обратной связи из issue **#182** — канал «**сотрудник**». Человек, который
обработал прайс-лист, прямо на странице результата может оценить итог (👍 / 👎 / 💡) и оставить
комментарий. Отзыв уходит на бэкенд и превращается в **GitHub issue** в настроенном репозитории.

Цель — собирать *структурированный* сигнал против реальных кейсов (а не анекдоты), чтобы итеративно
чинить два главных провала качества с боевого прогона: **сопоставление артикулов (vendorCode)** и
**определение РФ/РБ‑поставщика**.

> Документ для **операторов** (настройка интеграции) и **мейнтейнеров** (триаж issue). Два других
> канала #182 — **от MCP** (какие инструменты звались, no‑match/multi‑match) и **от агента**
> (агрегация `warnings[]` на `/metrics`) — это отдельные шаги, см. «Что дальше».

## Поток

```
UI (страница результата)
   └─ оценка + комментарий
        └─ POST /feedback   (сессия приложения + X-PAI-Auth, или Bearer)
             └─ backend/feedback.js  (санитизация + сборка issue)
                  └─ GitHub REST  POST /repos/<owner>/<repo>/issues
                       └─ issue с метками user-feedback, feedback:<kind>
```

Виджет рисуется только когда задание завершилось (`done`/`error`) **и** канал настроен на сервере —
UI заранее спрашивает `GET /feedback/config`.

## Конфигурация (ENV)

| Переменная | Обяз. | По умолчанию | Назначение |
|---|---|---|---|
| `GITHUB_FEEDBACK_TOKEN` | да* | — | GitHub‑токен с правом **Issues: write** на целевом репо. Нет токена → канал ВЫКЛЮЧЕН (виджет скрыт, `POST /feedback` → 503). |
| `GITHUB_FEEDBACK_REPO` | нет | `postroyka/purchase-ai-chat` | Куда писать issue, в формате `owner/repo`. Валидируется по `^[A-Za-z0-9_.-]+/[A-Za-z0-9_.-]+$` до запроса. |
| `FEEDBACK_RATE_LIMIT_MAX` | нет | `5` | Сколько отзывов с одного клиента за окно (антиспам в наш репозиторий). |
| `FEEDBACK_RATE_LIMIT_WINDOW_MS` | нет | `3600000` (1 ч) | Длина окна rate‑limit. |

\* Обязателен только чтобы **включить** канал. Без него приложение работает как раньше, просто без формы
обратной связи.

### Токен

Рекомендуется **fine‑grained Personal Access Token**, ограниченный ОДНИМ репозиторием
(`GITHUB_FEEDBACK_REPO`), с единственным правом **Issues → Read and write**. Это минимальная привилегия:
утечка токена даёт максимум возможность писать issue в один репо, а не доступ к коду или другим репо.
Классический `repo`‑scope тоже подойдёт, но он шире — избегайте.

Метки `user-feedback`, `feedback:<kind>` GitHub создаёт автоматически при первом отзыве (серым цветом по
умолчанию). Хотите свои цвета — заведите их заранее в Settings → Labels.

## Форма issue

Успешная отправка создаёт issue:

- **Заголовок**: `[Обратная связь] <kind> · <первая строка комментария>` (обрезается до 120 символов,
  hostile‑символы вырезаны).
- **Метки**: всегда `user-feedback` и `feedback:<kind>`, где `<kind>` ∈ `positive` | `problem` | `suggestion`.
- **Тело**: блок «Контекст» (jobId, имя файла, № сделки, кто сообщил, версия сборки, User‑Agent — только
  заполненные поля) + комментарий сотрудника внутри `<pre><code>` (HTML экранирован, Markdown инертен).
  Подвал помечает программное происхождение.

## Эндпоинты

### `GET /feedback/config` — открыт

```json
{ "enabled": true }
```

Возвращает ровно один булев флаг — настроен ли канал. **Ни токен, ни репо не раскрываются.** Нужен UI,
чтобы решить, показывать ли виджет.

### `POST /feedback` — требует авторизации (сессия приложения + `X-PAI-Auth`, либо Bearer)

Запрос:

```jsonc
{
  "kind": "problem",                // 'positive' | 'problem' | 'suggestion' (обязательно)
  "comment": "Артикул не найден...",// непустая строка, до 5000 симв. (длиннее — обрезается)
  "context": {                      // всё опционально; собирается приложением
    "jobId": "abc-123",             // ^[A-Za-z0-9-]{1,64}$, иначе отбрасывается
    "fileName": "price.xlsx",
    "dealId": "42",                 // ^\d{1,12}$, иначе отбрасывается
    "appVersion": "<git sha>"
  }
}
```

Ответ `201`:

```json
{ "ok": true, "url": "https://github.com/owner/repo/issues/7", "number": 7 }
```

Коды:

| Код | Когда |
|---|---|
| `201` | Issue создан. |
| `400` | Неизвестный `kind` или пустой `comment`. |
| `401` | Нет валидной авторизации. |
| `429` | Превышен rate‑limit (см. ENV). |
| `502` | GitHub отклонил/недоступен (детали — только в логах сервера, без токена). |
| `503` | Канал не настроен (`GITHUB_FEEDBACK_TOKEN` пуст). |

`url` указывает на issue в **приватном** репо — он не показывается рядовому сотруднику (тот видит только
тост «Спасибо»); это сигнал для приложения/мейнтейнера.

## Безопасность

Модуль `backend/feedback.js` портирован из проверенного `mcp/server/utils/github-feedback.ts` и
сохраняет его свойства:

- **Токен не попадает в логи** — ни сам токен, ни URL запроса, ни тело ответа GitHub не пишутся при
  ошибке (оператор из логов контейнера не восстановит креденшл). Логируется только стабильный код
  ошибки (`NOT_CONFIGURED` / `UPSTREAM` / `NETWORK`).
- **Валидация репо** — slug `owner/repo` проверяется до попадания в путь запроса (значение вроде
  `../../users/x` не уведёт вызов на чужой ресурс).
- **Вырезание hostile‑символов** — C0‑контролы, bidi‑оверрайды (U+202A..U+202E, U+2066..U+2069),
  zero‑width/BOM удаляются: подделанный комментарий не «перевернёт» список issue (Trojan Source) и не
  сломает JSON.
- **Инертный рендер** — свободный текст HTML‑экранируется и оборачивается в `<pre><code>`, так что
  Markdown/HTML в нём не исполняется.
- **Авторизация** — `POST /feedback` за `requireAuth`: только пользователи приложения (cookie‑сессия +
  `X-PAI-Auth`, либо серверный Bearer) могут заводить issue, иначе это открытый спам‑вектор в наш репо.
- **Rate‑limit** — окно считает **попытки**, а не успехи: неудачный вызов GitHub тоже тратит слот, что
  гасит тугие циклы ретраев. Тот же Redis‑лимитер, что у `/upload` (мультиинстансо‑безопасный).
- **Таймаут + `redirect: 'error'`** на вызове GitHub — зависший GitHub не «прибьёт» запрос, а 3xx на
  неожиданный адрес не выполняется (defence in depth).

## Приватность — почему контекст допустим

Целевой репозиторий **приватный на запуске**. Поэтому в issue допустимо складывать контекст задания
(jobId, имя файла, № сделки, кто сообщил) — это ровно тот сигнал, по которому мейнтейнер воспроизводит
кейс «артикул не сопоставился / поставщик не РФ». Имя файла и комментарий могут содержать данные клиента
— это сознательный компромисс для приватного репо.

> ⚠️ **Если репозиторий когда‑либо станет публичным** — пересмотрите это: каждое тело issue станет
> world‑readable. Тогда либо заведите отдельный приватный репо под `GITHUB_FEEDBACK_REPO`, либо урежьте
> контекст. Это отличие от агентского канала MCP, который пишет в *публичный* шаблонный репо и потому
> запрещает любые персональные данные (`mcp/docs/FEEDBACK.md`).

## Триаж

- Новые issue падают с меткой `user-feedback` в открытый бэклог — просматривать хотя бы раз в неделю.
- Перевешивайте на `bug` / `enhancement` / `wontfix` / `duplicate`, но **оставляйте** `user-feedback`,
  чтобы канал оставался queryable.
- Группируйте по корню (артикулы, РФ/РБ), а не закрывайте дубли — ссылайтесь на исходный issue.
- `jobId` в теле позволяет (пока не сработала ретенция `UPLOADS_RETENTION_DAYS`) поднять исходные файлы и
  результат агента на сервере.
