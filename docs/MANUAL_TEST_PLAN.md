# План ручного тестирования — procure-ai

Чек-лист приёмочной и портал-проверки: то, что **нельзя** покрыть автотестами и нужно прогнать
руками на тестовом портале Битрикс24 перед тиражом. Автотесты (`ui` vitest, `backend` vitest,
`b24-controller` PHPUnit, MCP-overlay) проверяют логику в изоляции; здесь — поведение целиком,
от загрузки файла до сделки в реальном портале.

Формат: **сценарий → шаги → ожидаемый результат**. Колонку «✓/✗» заполняйте при прогоне.

> Связанные доки: установка — [`BITRIX24_APP_SETUP.md`](BITRIX24_APP_SETUP.md); сервер —
> [`SERVER_SETUP.md`](SERVER_SETUP.md); сопровождение — [`OPERATIONS.md`](OPERATIONS.md); обратная
> связь — [`FEEDBACK.md`](FEEDBACK.md); бот — [`B24_BOT.md`](B24_BOT.md); бизнес-правила —
> [`PROJECT_BRIEF.md`](PROJECT_BRIEF.md).

## 4. Ядро пайплайна (главная функция)

| # | Сценарий | Шаги | Ожидаемо | ✓/✗ |
|---|---|---|---|---|
| 4.1 | **Все форматы** | Прогнать по одному файлу: PDF, XLSX, DOCX, XLS, JPG, PNG. | Текст извлечён (office → python-хелпер, PDF/скан → OCR), сделка создана. |  |
| 4.2 | **Поставщик по УНП** | Счёт от поставщика с УНП, заведённого в CRM. | Поставщик найден (реквизит `RQ_INN`, точное совпадение, min-id). |  |
| 4.3 | **Договор** | Поставщик с активным договором нужного типа. | Договор подобран (`procurecontract.find`), записан в `UF_CRM_DEAL_DOGOVOR`. |  |
| 4.4 | **Позиции/НДС** | Счёт с известными артикулами. | Позиции привязаны: цена БЕЗ НДС × количество, налог 20 %, флаг «НДС включён» = Y, единицы → штуки. |  |
| 4.5 | **Мультиматч (#195)** | Документ, где артикул/поставщик даёт >1 совпадения. | Берётся min-id; на `/metrics` растёт плитка «Мультиматчи». |  |
| 4.6 | **Артикул без матча (#195/#151)** | Позиция с артикулом не из каталога. | Позиция НЕ попадает в сделку; причина в отчёте; на `/metrics` — плитка «Артикулы без матча». |  |
| 4.7 | **Не-BYN** | Счёт в USD/EUR. | Сделка НЕ создаётся; на странице результата бейдж «Без сделки» + причина (#192). |  |
| 4.8 | **Карточка сделки** | Открыть созданную сделку в CRM. | Исходный файл во вложении (`UF_CRM_DEAL_SH_PRCHS_AI_FILE`); лог обработки в комментарии/таймлайне; заголовок «Импорт прайса от <поставщик>»; стадия `C1:NEW`, воронка «Закупки», валюта BYN. |  |
| 4.9 | **Пофайловый отчёт (#192)** | Загрузить пачку из «хорошего» и «проблемного» файлов. | По каждому файлу — свой статус: «Готово» / «Без сделки» + причина; у задания — «Частично». |  |
| 4.10 | **Лимит конкуренции** | Превысить `MAX_CONCURRENT_JOBS` одновременными загрузками. | Сверх лимита — `429` «Server busy». |  |

## 5. Обратная связь (#182 / #190)

| # | Сценарий | Шаги | Ожидаемо | ✓/✗ |
|---|---|---|---|---|
| 5.1 | **Отзыв сотрудника** | На результате нажать 👍/👎, добавить комментарий, отправить. | Тост «Спасибо за отзыв»; в приватном репо `GITHUB_FEEDBACK_REPO` появляется issue с контекстом (jobId/файл/сделка). |  |
| 5.2 | **Durable-outbox: сбой (#190)** | Временно закрыть egress к `api.github.com` (или сетевой сбой), отправить отзыв. | Ответ **202**, тост «Отзыв сохранён — отправим, как только GitHub станет доступен»; `GET /health` → `feedbackOutboxPending: 1`. |  |
| 5.3 | **Durable-outbox: дослыл (#190)** | Вернуть доступ к GitHub, подождать интервал (`FEEDBACK_OUTBOX_INTERVAL_MS`, по умолч. 60с). | Issue появляется в репо; `feedbackOutboxPending` → 0. Отзыв не потерян. |  |
| 5.4 | **Постоянная ошибка (#190)** | Указать несуществующий `GITHUB_FEEDBACK_REPO` (→ 404), отправить отзыв. | Ответ **502**, отзыв в очередь НЕ кладётся (оператор замечает по логу `UPSTREAM`). |  |
| 5.5 | **Приватность репо (#190)** | Временно сделать репо обратной связи публичным, перезапустить бэкенд. | В логах старта — WARNING «feedback repo … is PUBLIC». |  |
| 5.6 | **Канал не настроен** | Снять `GITHUB_FEEDBACK_TOKEN`. | Виджет отзыва скрыт; `POST /feedback` → `503`. |  |
