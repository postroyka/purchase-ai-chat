# Eval-набор агента (#93)

Замеряет **точность извлечения агента** на реальных счетах — то, чего не покрывают
unit-тесты (они проверяют код пайплайна, а не то, что модель правильно *прочитала*
документ). Закрывает риск №1 из оценки проекта: считает ли агент НДС по `÷1.2`,
матчит ли поставщика по УНП, отсекает ли не-BYN.

## Что где

| Файл | Роль | Где проверяется |
|---|---|---|
| `score.js` | чистая сверка результата агента с эталоном + направление НДС + агрегат (`summarize`) | `backend/tests/eval-score.test.js` (в обычном `pnpm test`) |
| `run.js` | прогон реального пайплайна по фикстурам → скоринг → агрегатная метрика | только на сервере (нужна модель) |
| `baseline.js` | прогон по счетам БЕЗ эталона → печать + черновик `<name>.expected.json` | только на сервере (нужна модель) |
| `scripts/samples/<name>.pdf` + `<name>.expected.json` | фикстуры + эталоны | — |

## Куда класть счета и эталоны

Всё в одной папке — `scripts/samples/` (или своя через `EVAL_SAMPLES_DIR`). На каждый счёт —
**два файла рядом, с одинаковым именем**:

```
scripts/samples/
  schet-001.pdf            ← сам счёт (PDF/XLSX/XLS/DOCX/PNG/JPG)
  schet-001.expected.json  ← эталон: что агент ДОЛЖЕН извлечь по этому счёту
```

`run.js` находит все `*.expected.json`, по каждому берёт соседний файл-счёт (поле `fixture`),
гоняет агента и сверяет. Не хочешь писать эталон руками — см. **Baseline** ниже.

## Запуск

```bash
make eval                      # = node backend/eval/run.js
```

⚠️ **Только там, где поднят агент**: нужны `claude` CLI, ключ модели
(`ANTHROPIC_*` / провайдер), доступный MCP-сервер (`MCP_SERVER_URL`,
`NUXT_MCP_AUTH_TOKEN`) и извлечение текста (`pdftotext`/OCR). В обычном CI не гоняем —
каждая фикстура это платный вызов модели. Логика скоринга при этом покрыта unit-тестами
и в CI бежит.

Переменные: `EVAL_SAMPLES_DIR`, `EVAL_RESPONSIBLE_ID`, `AGENT_TIMEOUT_MS`.

## Baseline: черновик эталона за тебя

Писать `expected.json` на каждый реальный счёт руками — долго. `make eval-baseline` гоняет
агента по счетам, у которых эталона ещё НЕТ, печатает, что он извлёк, и сохраняет **черновик**
`<name>.expected.json` из его вывода:

```bash
# 1. положи реальные счета в scripts/samples/ (пока без *.expected.json)
make eval-baseline
# 2. на каждый счёт появится <name>.expected.json с "draft": true
# 3. открой, сверь с документом, поправь поля, убери "draft": true,
#    впиши priceInclVatHint (цена С НДС из документа) — эталон готов
make eval     # теперь меряет точность
```

⚠️ Черновик — это **вывод агента**, а не истина: сравнивать агента с самим собой бессмысленно
(`run.js` предупредит про `"draft": true`). Польза baseline — не набивать структуру руками, а
сразу **проверять и править**. Окружение то же, что у `eval`. `EVAL_OVERWRITE=1` — перезаписать
существующие черновики.

## Безопасный прогон на реальных данных (тест-B24)

⚠️ Прогон eval запускает **полный конвейер**: агент **создаёт настоящие сделки** в том
Bitrix24, куда смотрит MCP (`MCP_SERVER_URL`). На реальных счетах через прод это засорит
**живой CRM**. Безопасный способ — изолированный стек `docker-compose.eval.yml`: поднимает
отдельный MCP, направленный на **тестовый** портал, и гоняет baseline против него. Боевые
`procure-app`/`procure-mcp` и живой портал не затрагиваются.

```bash
# 1. в .env.prod добавь вебхук ТЕСТ-портала:
#    NUXT_BITRIX24_TEST_WEBHOOK_URL=https://test-portal.bitrix24.ru/rest/1/<token>/
# 2. положи реальные счета в scripts/samples/ (без *.expected.json)
# 3. дай контейнеру (appuser) право писать черновики в эту папку:
chmod -R a+rwX scripts/samples
make eval-test     # baseline против ТЕСТ-портала → печать извлечённого + черновики
# 4. сверь черновики с документами, поправь, убери "draft": true → эталоны готовы
```

Внутри (`docker-compose.eval.yml`): свой проект `procure-eval` + сеть `eval-net`; `mcp-eval` с
переопределённым на тест вебхуком; `eval` (образ приложения, команда `baseline.js`) со свежим
**app-кодом** поверх образа — `backend/eval/`, `agent-runner.js`, `extract-text.js` и
`prompts/main.md` (иначе прогон шёл бы на коде устаревшего `:latest`-образа) — и счетами в
`/work/samples` (те же счета смонтированы и в `mcp-eval` — `create_deal` читает файл по пути).
После прогона стек гасится (`down`).

> Оверлей — **только app-сторона**. MCP-инструменты (`b24_pst_crm_*`) живут в `.output`
> mcp-образа и оверлеем не подменяются: при их отставании пересобрать mcp
> (`PUSH=0 make deploy-images`). И прогон проверяет **текущее рабочее дерево**, а не
> выложенный `:latest`-образ — это сигнал dev-цикла, не релизный гейт.

> Для **полного** теста со сверкой по эталонам поменяй в сервисе `eval` команду на
> `node /app/backend/eval/run.js` — когда черновики доведены до эталонов.

## Метрика (#93)

После прогона `run.js` печатает агрегат:
- **фикстуры** — сколько прошло целиком;
- **поля** — доля совпавших с эталоном проверок по всем фикстурам (общая точность извлечения);
- **направление НДС** — сколько позиций посчитано `÷1.2` против `×0.8` (баг #58) и процент
  ошибок: ключевой поведенческий сигнал.

```text
=== EVAL: 3/4 фикстур прошло ===
Поля: 27/31 совпало с эталоном (87.1%)
Направление НДС: ÷1.2 — 5, ×0.8 — 1  (ошибок ×0.8: 16.67%)
```

Агрегат считает `summarize()` в `score.js` — чистая функция, покрыта `eval-score.test.js`
(бежит в обычном CI; модель не нужна).

## Формат `<name>.expected.json`

Отказ (валюта/поставщик не прошли; договор НЕ блокирует — при отсутствии даёт warning `contract_not_found`, сделка создаётся):
```json
{ "fixture": "etalon-invoice.pdf", "expect": "error", "error": "foreign_supplier", "notes": "почему" }
```

Успешная сделка:
```json
{
  "fixture": "byn-ok.pdf",
  "expect": "deal",
  "currency": "BYN",
  "supplier": { "unp": "191234567" },
  "items": [
    { "name": "Цемент М500", "priceExclVat": 100.0, "quantity": 10, "priceInclVatHint": 120.0 }
  ]
}
```

- `priceInclVatHint` (опц.) — цена С НДС из документа. Скорер проверит, что агент
  поделил на 1.2, а не умножил на 0.8 (баг #58). Если в документе цена и так без НДС —
  не задавать.
- Позиции сверяются позиционно (в порядке документа); имя — нестрого.

## Текущие фикстуры

- **`etalon-invoice.pdf`** → `foreign_supplier`. Это счёт **российского** поставщика
  (ООО «ЭКОДИЗАЙН», ИНН 10 цифр + КПП, ПАО Сбербанк Москва) в **RUB**. Корректное
  поведение — ОТКАЗ по **реквизитам поставщика** (КПП/ИНН ⇒ РФ), а не по валюте (#97).
  Ловит подвохи: «руб.» не различает RUB/BYN; «(Россия)» в названии товара — страна
  происхождения товара, а не поставщик.
